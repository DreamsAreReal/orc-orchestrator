# RS-00b — Брейншторм: максималист

**Угол:** «что сделало бы автономный контур ВЫДАЮЩИМСЯ, а не просто рабочим?»
**Вход:** RS-01 (волна A, 55 находок), STATE.md, архитектура beads + диспетчер + интерактивные терминалы + Happy.
**Дата:** 2026-07-15. Режим: брейншторм (вердикта нет).

---

## Ключевое наблюдение угла

Рабочий контур = «задачи выполняются ночью». Выдающийся контур определяется ДВУМЯ
моментами, которых в RS-01 нет как продуктовых сущностей:

1. **Момент пробуждения** — единственная точка, где пользователь ПРОЖИВАЕТ ценность.
   Сейчас траектория потребления утра не деконструирована: пользователь получает
   10 терминалов с историей и сам разгребает, что произошло. Это анти-вау.
2. **Латентность человека** — главный ботлнек всего контура. Машина параллелится,
   человек — нет. Все находки RS-01 о батчинге (Iqbal & Bailey −25% времени реакции)
   и HULA (82% планов утверждаются быстро) указывают на одно: рычаг максимального
   порядка — не ускорять агентов, а СЖИМАТЬ и ПЕРЕМЕЩАТЬ моменты, где нужен человек.

Все идеи ниже — поверх зафиксированной архитектуры, без её ревизии.

---

## Идеи (идея → почему выдающееся → цена)

### 1. «Утренняя газета» — ночная смена как деливерабл
**Идея.** К моменту пробуждения диспетчер собирает единый отчёт смены: по каждой
задаче — статус/дифф/артефакты, секция «инциденты ночи», секция «жду твоих решений»
(отсортирована — см. идею 8), сожжённый бюджет окон vs прогноз. Одна дешёвая
haiku-сессия по STATE.md всех задач + git log; смена помечается git-тегом
`night-shift-YYYY-MM-DD`, отчёт показывает диффы «до/после смены» по проектам.
**Почему выдающееся.** Это и есть signature-механика продукта: «проснулся → прочитал
газету → поставил 4 визы → смена продолжилась». Опыт пробуждения перестаёт быть
разгребанием логов и становится ритуалом. Без этого весь контур невидим для
потребителя — ценность есть, а её ПЕРЕЖИВАНИЯ нет.
**Цена.** ~1 фича: генератор (шаблон + агрегация STATE.md/git), теги смен. Низкая.
Кандидат в signature-в-скелете (F1): скелет без «газеты» не демонстрирует ценность.

### 2. Inbox решений: все гейты — карточками в Happy
**Идея.** AskUserQuestion всех задач стекаются в единый упорядоченный inbox: каждая
карточка = контекст на один экран телефона (ТЗ-дельта, топ-3 риска, рекомендация
агента + почему), варианты — кнопками. Пользователь «свайпает» 10 решений за
5 минут за кофе. Доставка — батчем к границам задач (Iqbal & Bailey, TOCHI:
−20% фрустрации, −25% времени реакции), не синхронными дёрганиями.
**Почему выдающееся.** Сокращение цикла «вопрос→ответ» с часов до минут умножает
throughput сильнее любого параллелизма инстансов: припаркованная задача не жжёт
токены, но жжёт календарное время и окна. HULA подтверждает: 82% планов
утверждаются быстро — значит формат «карточка+кнопки» покрывает подавляющее
большинство гейтов без открытия ноутбука.
**Цена.** Конвенция формата вопроса в промпте воркера (дёшево) + возможно тонкий
агрегатор поверх Happy. Риск: Гудхарт на карточку — агент сжимает контекст так,
что решение принимается вслепую → гейту нужен пункт «карточка обязана содержать
ссылку на полное ТЗ, и цена ошибки решения названа явно».

### 3. Спекулятивное предисполнение веток гейта
**Идея.** Задача запаркована на гейте ТЗ, окна простаивают → диспетчер запускает
build по варианту-рекомендации в отдельном worktree с пометкой SPECULATIVE.
Утром «да» = работа уже готова; «нет» = worktree выбрасывается без следа в main.
Инвариант: спекулятивная ветка НИКОГДА не мержится без визы человека.
**Почему выдающееся.** Экономика уникально благоприятна и вся выведена из RS-01:
лимиты недоиспользуются (окна всё равно сгорают впустую), припаркованные задачи
не тратят пул, HULA даёт ~82% вероятность утверждения рекомендации → EV
спекуляции резко положительный. Пользователь утром получает не «жду решения»,
а «уже сделано — подтверди». Это качественный скачок ощущения автономии.
**Цена.** Worktree-изоляция (ADR по worktree уже запланирован), правило
не-мержа, учёт в планировщике окон. ~18% работы уходит в корзину — приемлемо
при недоиспользуемых лимитах. Уже лежит в backlog STATE.md — поднять из P2 в ТЗ.

### 4. Yield-management окон вместо реактивного back-pressure
**Идея.** Диспетчер знает времена ресета (5-часовое окно, weekly caps, Opus-кап —
детерминированные строки CLI) → планирует ночь как bin-packing: тяжёлые
Opus-фазы (ТЗ, design) — к свежим окнам; verify/lint/доводка — на хвосты окон
перед ресетом, чтобы остаток не пропадал; «догрузить мелочь из backlog» как
наполнитель хвоста.
**Почему выдающееся.** Превращает back-pressure (реагируем на исчерпание) в
проактивное управление доходностью. Продукт затевался ради «лимиты
недоиспользуются» — это единственная идея, которая прямо атакует корневую
метрику: % утилизации оплаченного пула за смену.
**Цена.** State-machine лимитов уже пишется — сверху прогноз и очередь с
классами фаз. Риск: проактивного API остатка нет (принятая дырка RS-01) →
эвристики по ccusage + реактивная коррекция; закладывать как «планировщик
советует, back-pressure остаётся жёсткой стеной».

### 5. Model-tiering по фазам конвейера
**Идея.** Деградацию «Opus-кап → Sonnet» из RS-01 сделать не аварийным
фолбэком, а нормой планирования: маппинг фаза→модель в конфиге (ресерч-верификация
ссылок, характеризационные прогоны, janitor, «газета» — Haiku/Sonnet; ТЗ, design,
сложный build — Opus). Оценка сложности задачи — метка в beads при закидке.
**Почему выдающееся.** Умножает эффективное число задач за смену в разы при том
же пуле: Opus-кап — самый дефицитный ресурс, и сейчас он тратится на работу,
которую Haiku делает неотличимо. Вместе с идеей 4 даёт «дважды выжатый» пул.
**Цена.** Конфиг + передача модели при запуске воркера. Риск падения качества
на пограничных фазах → эвалюатор конвейера как стена; правило эскалации
«две неудачи на Sonnet → повтор на Opus».

### 6. Чёрный ящик watchdog-а + утреннее ретро инцидентов
**Идея.** Каждый kill (петля по hash, тишина heartbeat) записывает крэш-досье:
последние N heartbeat, хвост транскрипта, git diff на момент смерти, вердикт
детектора — в `docs/tasks/<слаг>/incidents/`. «Газета» показывает секцию
инцидентов; повторяющийся паттерн (≥2 раза) автоматически становится
кандидатом в `references/failures.md` конвейера через ретро.
**Почему выдающееся.** Контур начинает УЧИТЬСЯ на собственных ночных провалах —
единственная идея, дающая компаундинг: через месяц эксплуатации у конвейера
корпус реальных (не вычитанных) провалов. Свежий рестарт от STATE.md (2509.09677)
остаётся механикой восстановления, но перестаёт быть амнезией.
**Цена.** Почти нулевая: heartbeat-хук уже пишет всё нужное; добавить сборку
досье при kill + строку в ретро-протокол.

### 7. Canary-предполёт перед сменой
**Идея.** За 3 минуты до старта смены — смок: `claude auth status` (Keychain жив),
`bd ready --json` парсится, терминал открывается osascript-ом, ccusage отвечает,
Happy доставляет тест-push «смена начинается: N задач, прогноз M окон».
Любой fail → громкий push, ПОКА пользователь ещё не спит.
**Почему выдающееся.** Убивает главный анти-вау сценарий: «проснулся — ничего не
работало из-за протухшего токена» (Keychain — риск №1 из карты рисков STATE.md).
Надёжность первой ночи = судьба доверия к продукту; один тихий провал смены —
и пользователь возвращается к ручному режиму. Push «смена начинается» — сам по
себе signature-момент (ритуал передачи смены машине).
**Цена.** Копеечная: один скрипт из уже выбранных проб исполнимости. Обязан
попасть в скелет F1 — это буквально проба исполнимости, переиспользованная как
прод-фича.

### 8. Сортировка гейтов по «блокирующей мощности»
**Идея.** Очередь решений (идея 2) сортируется не FIFO, а по цене простоя:
сколько задач/окон разблокирует ответ — считается топологически по графу
зависимостей beads (`bd dep`), бесплатно доступному диспетчеру. Карточка
показывает: «этот ответ разблокирует 3 задачи и ~2 окна ночи».
**Почему выдающееся.** 30 секунд внимания пользователя тратятся там, где
разблокируется максимум машинного времени; человек начинает видеть себя
диспетчером фабрики, а не отвечателем на вопросы. Дешёвая идея с
непропорционально сильным эффектом на утилизацию.
**Цена.** Минимальная: подсчёт по графу + поле в карточке.

### 9. Autonomy dial: уровень поводка per-task
**Идея.** При закидке задача получает метку уровня автономии: L0 — гейт на всё;
L1 — стандарт (гейт ТЗ); L2 — «доверенный тип»: без гейта ТЗ, парковка только
по аномалиям (аномально быстрый зелёный, деструктивные операции, отклонение от
ТЗ-дельты). Маппинг на уже существующие режимы конвейера (микро/lite/полный).
**Почему выдающееся.** Превращает автономию из бинарной в управляемый градиент.
Рутинные доверенные классы (обновить зависимости, прогнать перевод, сгенерить
отчёт) едут ночью ВООБЩЕ без человека — это и есть обещание «накидал 10 задач —
Mac сам довёл», честно выполненное хотя бы для части классов с первого дня.
**Цена.** Метка в beads + маппинг. Риск серьёзный: L2 — открытая дверь для
reward hacking (METR: промпт-запреты бесполезны) → L2 легален ТОЛЬКО при
внешних стенах из RS-01 (read-only оракулы, deny-rules, sandbox, гейт на
аномально быстрый зелёный) — зафиксировать в ТЗ как инвариант, не опцию.

### 10. Merge queue + janitor (паттерн Bernstein)
**Идея.** DONE-задачи не мержатся сразу: попадают в merge queue проекта;
дешёвая janitor-сессия перепрогоняет тесты на актуальном main и мержит по
одному; конфликт/красное → задача возвращается в ready с автозаданием rebase.
**Почему выдающееся.** Закрывает главный источник утреннего разочарования при
параллели: «зелёное по отдельности — красное вместе». Без этого worktree-параллель
внутри проекта (опция из требований пользователя) небезопасна в принципе.
Bernstein в RS-01 назван донором именно этих паттернов — идея просто соединяет
уже подтверждённую находку с уже заявленной опцией, чего никто не сделал.
**Цена.** Средняя: очередь + janitor-промпт; janitor на Haiku (идея 5) почти
бесплатен по пулу.

### 11. Инверсия расписания: человеко-фазы — днём, машинные — ночью
**Идея.** Сразу после утренней закидки дешёвый агент делает «прогрев» каждой
задачи: фаза 0 + ресерч-lite + ПОДГОТОВКА ВОПРОСОВ гейта — и задаёт их
пользователю днём, пока тот у компа/телефона. Ночь остаётся чисто машинной:
build/verify без единого гейта. Диспетчер различает «фазы, требующие человека»
(планировать в окно присутствия) и «чисто машинные» (планировать в ночь).
**Почему выдающееся.** Это ключевая инверсия: все идеи 1–3 борются с латентностью
человека ночью, а эта её УСТРАНЯЕТ — гейты перестают попадать в ночь вообще.
В пределе смена не паркуется ни разу. Синергия с идеей 4 (человеко-фазы дёшевы
по Opus — им дневные остатки окон).
**Цена.** Планировщику нужна классификация фаз по «нужен ли человек» (она уже
есть в стейт-машине конвейера — гейт ТЗ известен заранее); UX-риск: дневные
вопросы не должны превращаться в синхронные дёргания — батчить (идея 2).

### 12. Телеметрия смены как продукт: график «окна × задачи»
**Идея.** OTel-находку RS-01 (CLAUDE_CODE_ENABLE_TELEMETRY, дашборды
ColeMurray/claude-code-otel) включить не «опционально», а как источник
одного графика в «газете»: горизонталь — ночь, дорожки — задачи, заливка —
сожжённые окна/модели, флажки — инциденты и гейты.
**Почему выдающееся.** Единственный способ, которым пользователь ФИЗИЧЕСКИ
увидит работу ночной фабрики (терминалы закрыты, ночь прошла). Доверие к
автономии строится на наблюдаемости; график превращает абстрактное «лимиты
использованы лучше» в наглядную картинку — и даёт данные для тюнинга идей 4–5.
**Цена.** Низкая-средняя: env-переменная + готовый дашборд как донор; рендер
в «газету» — самое дорогое, можно начать с ASCII/таблицы.

## Следствия для ТЗ (сводно)

**Требования-кандидаты (P0/P1):**
- «Утренняя газета» + canary-предполёт — в скелет F1 (это траектория потребления
  и надёжность первого впечатления; без них продукт не переживается).
- Формат карточки гейта (контекст одним экраном + цена ошибки + ссылка на полное
  ТЗ) — конвенция промпта воркера, P0.
- Сортировка гейтов по блокирующей мощности — P1, дёшево.
- Model-tiering (фаза→модель) и классификация фаз «нужен человек / машинная» —
  поля конфига диспетчера с первого дня, даже если планировщик наивный.

**Требования-кандидаты (P1/P2, отдельные фичи):**
- Спекулятивное предисполнение (поднять из backlog; инвариант не-мержа без визы).
- Yield-management окон (поверх state-machine лимитов).
- Merge queue + janitor (обязателен ДО включения worktree-параллели в проекте).
- Крэш-досье + путь «инцидент → кандидат в failures.md» в ретро-протоколе.
- Autonomy dial L0–L2 (L2 только при внешних стенах — инвариант, не опция).
- OTel-график смены в «газете».

**Гейты/метрики-кандидаты:**
- Утилизация пула за смену (% использованных окон) — главная метрика продукта.
- Время цикла «вопрос→ответ человека» (медиана) — метрика inbox-а решений.
- «Ноль тихих провалов»: смена либо отработала, либо пользователь получил push
  ДО сна (canary) или в момент смерти воркера — проверяемо по журналу.
- Доля ночных гейтов → к нулю (метрика инверсии расписания, идея 11).

**Риски, добавляемые идеями:**
- Гудхарт на карточку гейта (решение вслепую) — идея 2.
- Спекулятивная ветка в main без визы — идея 3 (инвариант не-мержа).
- L2-автономия без внешних стен = приглашение reward hacking — идея 9.
- Планировщик окон, доверяющий эвристике остатка квоты, — идея 4 (back-pressure
  остаётся жёсткой стеной поверх любых прогнозов).
