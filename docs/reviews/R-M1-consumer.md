# R-M1 — Отчёт потребителя: CLI `orc`

## 1. Персона и продукт

**Что это (дословно из промпта):** «CLI-инструмент `orc`, который сам прогоняет твои
задачи через Claude Code и показывает статус.»

**Персона:** разработчик, который утром накидывает пару задач и уходит пить кофе, а
вернувшись — хочет понять, что произошло. Документацию читать не хочет.

**Как понял сам (до docs, только по `orc`/`--help`/интуиции):** очередь задач, которую
демон-«смена» разбирает, спавня терминалы с Claude Code; `status` — пульт наблюдения,
газета (`--newspaper`) — сводка «что случилось за смену». Модель `init → add → start →
status` считалась с первого экрана без документации.

## 2. Журнал потребления (шаги → что увидел → где споткнулся)

1. `bin/orc` (без аргументов) → usage с 4 командами и описаниями в скобках. **Порядок
   действий понятен без docs.** ~2 сек.
2. `bin/orc <cmd> --help` для всех 4 → `add` и `start` описаны в скобках хорошо
   (`--once` = «spawn a single worker and return»). **Споткнулся:** `init --help` не
   имеет ни описания команды, ни описания флага `--json` — из help непонятно, что
   `init` делает и, главное, ГДЕ создаёт хаб.
3. Создал тест-проект `~/Desktop/orc-consumer-test` + `git init` + первый коммит.
4. `add` ДО `init` (нетерпеливый путь) → чёткая ошибка: «orc hub is not initialized.
   Run `orc init` first (creates .beads queue)». **Подсказка спасает.** Но обещает
   «.beads queue».
5. `orc init` → «orc hub initialized at **/Users/admin/.orc**». **Споткнулся:** хаб
   создан ГЛОБАЛЬНО в `~/.orc`, не в проекте и не в cwd. Плюс терминологический
   разнобой: ошибка звала это «.beads», сообщение — «~/.orc» (по факту внутри
   `~/.orc/.beads`). Первому пользователю неочевидно, что хаб один на все проекты.
6. `orc add ~/…/orc-consumer-test "Create a file HELLO.txt containing the single word:
   hi"` → «added task orc-xxa -> …». OK.
7. `orc status` сразу после add → **«смена не запущена. Поставь задачи (`orc add`) и
   запусти (`orc start`).»** **Споткнулся (главное):** только что добавленная задача в
   status НЕ ВИДНА вообще — вывод говорит «поставь задачи», будто очередь пуста.
   Пользователь может решить, что задача потерялась. Спасает лишь то, что подсказка
   явно велит `orc start`.
8. `orc start --once` → панель `=== canary preflight ===` со списком `[ ok ]` (bd,
   auth, ccusage «56 мин осталось», notify, ram «1743 MB free», spawn) → «spawned
   worker for orc-xxa … **(pid None)**». **Споткнулся:** `pid None` выглядит как сбой
   спавна — обещали воркера, а PID пустой.
9. Проверка `ps` → воркер РЕАЛЬНО живёт (claude с промптом «…Task: Create a file
   HELLO.txt…»). Значит `pid None` — не сбой: `status --json` показал, что orc
   трекает воркера по вкладке терминала (`"session":"tab 1 of window id 4401"`,
   `"pid": null`), PID он вообще не знает.
10. `orc status` (live) → читается хорошо: `▸ orc-xxa build active 0м` + строка
    ресурсов «пул: 81% окна, 56 мин осталось, RAM ok». **Это тот экран, что нужен
    вернувшемуся за кофе.**
11. Поллинг ~4 мин. На 2-й минуте в проекте появились `docs/tasks/…/STATE.md` и
    `docs/HELLO.txt` (`hi`); STATE.md: «Фаза 5 VERIFY → DONE, Статус: **DONE**».
    **Предметная задача выполнена.**
12. `orc status --newspaper` в тот же момент и далее до 4-й минуты → «смена: **0
    готово**, 0 ждут тебя, 0 упало» + задача всё ещё «**в работе / build active**».
    **Споткнулся (критично для персоны):** газета говорит «ничего не готово, всё ещё
    в работе», хотя файл создан и STATE=DONE. Воркер-процесс висит после завершения
    предметной работы, orc держит вкладку «активной» → задача не переходит в «готово».

## 3. Стресс-режим (беглое потребление)

По одному `orc` без аргументов + первым словам вывода собирается верная ментальная
модель: очередь → смена → статус/газета. Preflight-панель и строка ресурсов
самообъясняемы. Слабое звено при беглом чтении — состояние «между add и start» (status
делает вид, что очередь пуста) и «после фактического DONE» (газета не догоняет).

## 4. Стандартная рубрика (факты)

- **Время/действия до первого значимого результата:** ~5 команд (`orc` → helpы → init
  → add → start), затем ~2 мин ожидания до появления `docs/HELLO.txt` с верным `hi`.
  Реальный путь включал 1 ошибочную ветку (`add` до `init`).
- **Счётчик повторов:** максимальная серия одинаковых действий без новой информации —
  **~9 подряд** вызовов `orc status --newspaper` (шаги 12+), каждый возвращал тот же
  «0 готово / build active», хотя задача уже была DONE. Новой информации не появлялось.
- **Последнее осмысленное решение:** «задача выполнена — можно забирать `docs/HELLO.txt`»
  — принято НЕ по газете orc, а по ручному `ls`/`cat` в проекте. Это находка: продукт
  своё ключевое обещание («показывает статус» → «пойми, что произошло») в моём прогоне
  не выполнил.
- **Момент ухода:** на шаге 12 (третий подряд «newspaper = 0 готово», а файл на диске
  уже есть) — вернувшийся пользователь поверил бы газете «ещё не готово», ушёл бы ждать
  дальше, не забрав готовый результат.
- **Что заставило бы вернуться завтра:** preflight-панель + однострочник ресурсов
  («56 мин осталось, RAM ok») — реально закрывают «ушёл пить кофе, не сжёг ли лимит».
- **Вау-вопрос (что показал бы/процитировал):** строку газеты-заголовка
  «`смена: N готово, M ждут тебя, K упало; съедено X% окна`» — она есть у меня в голове
  как формат сводки. Но показал бы с оговоркой: у меня она соврала (0 готово при
  фактическом DONE), так что «вау» пока авансом на формат, не на достоверность.

**Сырые счётчики (пороги в промпте не заданы — вердикт по ним не выношу):** различных
механик за первые 5 мин — 4 (init, add, start/preflight, status/newspaper); макс. серия
повторов без новой информации — ~9 (newspaper).

## 5. Топ-5 находок (место → факт → почему потеря)

1. **`status`/`newspaper` не догоняют фактический DONE.** Место: шаг 11–12. Файл
   `docs/HELLO.txt`=`hi` создан, STATE.md=DONE, а `--newspaper` держит «0 готово / в
   работе» ≥2 мин. → Прямой удар по обещанию продукта: вернувшийся за кофе поверит
   «не готово» и не заберёт готовый результат. Причина в трекинге по вкладке
   терминала, а не по завершению работы/STATE.
2. **После `add` задача невидима в `status`.** Место: шаг 7. `status` печатает «смена
   не запущена. Поставь задачи…», будто очередь пуста, хотя `orc-xxa` уже добавлена.
   → Пользователь думает, что задача потерялась; нет команды «покажи очередь до
   старта».
3. **`spawned worker … (pid None)` выглядит как сбой.** Место: шаг 8. PID пустой,
   потому что orc трекает вкладку, а не процесс — но CLI это не объясняет. → Ложный
   сигнал тревоги при штатном запуске.
4. **`init` молча создаёт ГЛОБАЛЬНЫЙ хаб в `~/.orc`.** Место: шаг 5. Один хаб на все
   проекты, не в проекте; `init --help` без описания, ошибка зовёт это «.beads», а
   сообщение — «~/.orc». → Неочевидная модель «где живут мои задачи»;
   терминологический разнобой .beads/.orc сбивает.
5. **Смешанный язык вывода.** Место: сквозной. `add`/`start`/preflight — английский
   (`added task`, `spawned worker`, `[ ok ] auth`), а `status`/газета — русский
   (`смена не запущена`, `в работе`, `ждут тебя`). → Разнобой ломает восприятие
   «единого пульта», особенно при беглом чтении.

## 6. Итог

Основную ценность продукт в моём прогоне отдал ЧАСТИЧНО: предметная задача выполнена
корректно (`docs/HELLO.txt` = `hi`, STATE=DONE) и запуск/preflight отработали прозрачно
— **но центральное обещание «показывает статус → пойми, что произошло» провалилось**:
в момент фактического DONE и после него `status`/`newspaper` показывали «0 готово / в
работе», и как пользователь я узнал результат только ручным `ls`/`cat`, в обход orc.
Для целевой персоны (вернулся за кофе, читает газету, а не диск) это отказ по главному
сценарию.

**ЧЕКПОЙНТ: ПРОВАЛЕН** — место: шаги 11–12, рассинхрон «STATE=DONE / файл на диске» vs
«newspaper: 0 готово, задача в работе»; вернувшийся пользователь по единственному
предусмотренному каналу (газета) не поймёт, что задача завершена, и не заберёт результат.
