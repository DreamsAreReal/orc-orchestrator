## 1. Профминимум и базовые гейты (quality-baselines)

G0a (узкий скоуп × 100%), G0b (реально работает, не display-only), G0c (секреты/PII,
хук) — перенесены ПЕРВЫМИ строками таблицы ворот, с метрикой/способом/порогом. Зачёт.
Дисциплина измерений: базлайн improvement-части есть (doctor exit 0 замерен, docs/tasks
0 точек по RS-02, характеризационный набор scorecard фиксируется ДО патча); гейты G9 —
дельты от него, регрессий 0. Принцип 15 соблюдён. Контр-метрики (guardrails) — таблица
из трёх пар, включая анти-rubber-stamp. Зачёт.

## 2. Критерии-исходы и пороги

Все G0–G11 — исходы с числовыми порогами, зафиксированными до работ («3/3», «0 дублей»,
«≤10 сек», «≤5 мин», «100% фикстур»). Критериев-«действий» не обнаружено. Субъективные
слова запроса («по умному», «сам довёл») вынесены в прокси-таблицу — рецепт
class-profiles исполнен. НО в прокси-таблице сломана ссылка: «G3 сериализация + **G5
допуск**» — G5 в Воротах = Watchdog, а гейта на критерий «задачи не простаивают при
свободном окне/RAM >15 мин» в таблице ворот НЕТ ВООБЩЕ. Критерий «по умному» — ядро
обещания утилизации пула — остаётся без измеряющего гейта (СУЩ).

## 3. Тест Гудхарта по гейтам

- G1 (E2E 3/3): закрыт внешними фактами (git/артефакты, аудит пост-условий) + контр-метрика
  «verify-фаза полная» — скриптовый оптимум «DONE заявлением» перекрыт. Зачёт.
- G2 (гейт-протокол): «потерь контекста 0» — неоперационализировано (как мерить потерю?
  нужен наблюдаемый признак: продолженная сессия не переспрашивает решённое /
  ссылается на утверждённое ТЗ) — МЕЛОЧЬ.
- G5 (watchdog): «детект ≤ порога конфига» — самоссылка: гейт проходится на удобном
  тестовом конфиге и не привязывает продовый дефолт; нужен консервативный дефолт в ТЗ —
  МЕЛОЧЬ. Ложные kill = 0 при живом длинном tool — правильная контр-проверка. Зачёт по сути.
- G11 (постановка ≤5 мин): не сказано, что меряется ЖИВОЙ пользователь — скрипт из stdin
  даёт 10 секунд, не доказывая UX утреннего оператора (RS-00 угол 1) — МЕЛОЧЬ.
- **Гейт-карточка**: собственный ресерч (RS-00b идея 2, Р11) назвал Гудхарт «агент сжал
  контекст → решение вслепую» и лекарство: карточка обязана нести ссылку на полное ТЗ
  и явную цену ошибки. Карточка brief = «3 строки: скоуп/планка/полномочия» — лекарство
  выпало; кап батча (≤5) и «необратимое — вне батч-аппрува» тоже не зафиксированы.
  Контр-метрика «время-до-аппрува» ловит быстрый тап, но не слепую карточку. СУЩ.

## 4. Жанровая полнота против собственного ресерча (следствия RS-00c)

Покрыто: стены против reward hacking (внешние оракулы, G1), watchdog с внешней
проверкой пост-условий (Р2/П6), бюджет-капы до замера (Р3, условия R-research р.2
все три исполнены), патчи ~/.claude отдельным майлстоуном с канарейкой и откатом (Р4),
pin CLI + self-test на фикстурах (Р7, G4), дневной контур с ночным P1 (Р6, П2),
tmux P1 (Р10), метрика времени-до-аппрува (Р11а).

НЕ покрыто и не отклонено ADR-ом:
1. **Р5 re-validate ТЗ-дельты при claim** («в ТЗ как явный шаг, не пожелание»). Сценарий
   реален именно в конфигурации brief: G2 требует «слот освобождён» при парковке →
   вторая задача того же проекта работает, пока первая ждёт аппрува → утверждённое ТЗ
   протухает к продолжению; G3 ловит пересечение интервалов, но не свежесть плана.
   Тихий семантический брак бьёт прямо в цель «без потери качества». СУЩ.
2. **Р8: человек и недобитые воркеры**. Pre-flight чистоты дерева (`git status
   --porcelain`), детект человеческой активности в репо, kill по факту смерти
   process-group — отсутствуют. Дневной контур (выбор П2!) делает «человек в репо во
   время смены» штатным, а не экзотикой; project-mutex человека не видит. СУЩ.
3. **Р10/TCC в canary**: предполёт проверяет bd/auth/ccusage/канал/RAM, но НЕ проверяет
   сам запуск воркера — osascript→Terminal из контекста LaunchAgent (TCC Automation
   спрашивается лениво, при первом использовании). Самый хрупкий механизм остаётся вне
   предполёта: смена стартует, воркеры молча не поднимаются, cap-2-рестарты жгут время.
   СУЩ.
4. Р3в (приоритет очереди от пользователя — «жечь Opus по приоритету»): `orc add` без
   поля приоритета (bd -p существует) — МЕЛОЧЬ.
5. Р6/Р9 dead-man switch «пульс контура» вторым каналом — для дневного v1 приемлемо
   опустить (пользователь рядом), фиксирую в ЧТО ПЛОХО как долг ночного режима.

## 5. Полномочия (фокус промпта: Р13 — bypass×$HOME, push, MCP)

- push: покрыт («ЗАПРЕЩЕНО без вопроса: git push (любой)»). Зачёт.
- **bypass×$HOME: НЕ покрыт.** Проба launchagent вскрыла глобальный alias
  `--dangerously-skip-permissions` — воркеры наследуют bypass на весь $HOME; RS-01:
  с v2.1.126 protected paths в bypass разрешены. В brief нет ни секции «границы
  песочницы», ни ограничений ЧТЕНИЯ (Полномочия регулируют только запись/действия),
  ни спайк-требования «sandbox/отдельный settings воркера ДО BUILD», ни письменного
  принятия риска. Условие снятия СУЩ р.2 не выполнено.
- **MCP: НЕ покрыт.** В окружении живут посторонние MCP (Glovo, shopping и пр.) —
  канал утечки контекста; MCP-allowlist per-воркер не упомянут.
- Enforcement: список «ЗАПРЕЩЕНО» не привязан к стенам (deny-rules/sandbox/hooks);
  собственный ресерч дважды фиксирует: промпт-запреты не работают (METR 80→70%).
Итого — БЛОКЕР (один, композитный: секция границ песочницы).
Отдельно: «установка dev-инструментов через brew/npm БЕЗ вопроса» прямо противоречит
Р14(3) («системные изменения — всегда человек») — если это осознанный грант, он должен
быть виден на гейте (СОМНЕНИЕ).

## 6. Ставка вау, паспорт, петля потребления

- Ставка: ось ЕДИНСТВЕННАЯ и конкретная (опыт «утренней газеты»), ставочные фичи
  названы (shift report, гейт-карточки, canary), риск скуки с контрходом (первый экран
  ≤150 слов). Зачёт по 9a.
- Паспорт: N/A на визуал легален по профилю (CLI без UI) и КОМПЕНСИРОВАН словарём
  класса: дисциплина вывода конкретна (plain text, первая строка — важное, ≤80 колонок,
  --json у каждой команды, анти-слоп «Successfully!»), референс-эталон назван (bd CLI +
  Bernstein, деконструкция в appendix), signature-элемент с примером строки итога.
  Зачёт по 11.
- Петля: постановка ≤5 мин → смена → гейт-карточки → `orc status`-газета — такты есть,
  прокси и метрики к ним привязаны. Слабости: (а) Цель и ставка сформулированы под
  «утро/проснулся», тогда как v1 по П2 — ДНЕВНОЙ контур (ночь = P1/вопрос гейта) —
  формулировки прообещают режим, который не в P0 (МЕЛОЧЬ); (б) середина смены не
  специфицирована: что показывает `orc status`, пока смена ИДЁТ (сейчас работает X,
  в очереди Y, ждут гейта Z) — газета описана только как итог (ЧТО ПЛОХО, риск к design).
- Signature-в-скелете: brief не фиксирует, что скелет F1 обязан содержать газету+canary
  (RS-00b прямо предлагал) — оставляю рекомендацией design-фазе (ЧТО ПЛОХО).

## 8. Цена среза (cut-list к гейту; решение — за пользователем)

| Срез | Цена в ущербе потребителя контура |
|---|---|
| Гейт-карточки (голый список ждущих) | rubber-stamp с 1-го дня; решения вслепую; ось ступени 4 умирает — продукт вырождается в cron с логами |
| Canary-предполёт | одна протухшая auth = смена мертва молча, узнаёшь вечером; −1 день доверия (анти-вау №1 по Р9) |
| Патчи конвейера (docs/tasks/) | мультизадачность в проекте невозможна → смена ≤ N проектов задач; «10 задач по 2-3 проектам» не собирается в принципе |
| G6 восстановление | любой крэш диспетчера = ручная реанимация + риск дублей (два пишущих в репо) |
| G4 self-test фикстур лимитов | первый апдейт CLI = слепой диспетчер; риск сжечь weekly-кап за утро (окно — валюта продукта) |

  спайк-подтверждением ДО BUILD либо письменное принятие риска») + принцип 7 (условная
  деградация не снята) + Р13/Р14 (промпт-запреты не работают, METR). — Фикс: секция

  неизмерим (принцип 3/14). — Фикс: добавить гейт допуска/утилизации (лог диспетчера

## ЧТО ПЛОХО (раунд 3)

- **Опыт:** live-`orc status` обещает «минуты/расход» per-задача, но источник
  per-задачной атрибуции расхода нетривиален (ccusage меряет ГЛОБАЛЬНЫЙ пул) — если
  design не решит атрибуцию честно, колонка станет приблизительной и подорвёт доверие
  к строке итога; референс (bd) показывает только то, что знает точно.
- Газета не обещает «что смена НЕ взяла и почему» (задачи, не влезшие в окно/RAM) —
  НЕ конвертирую в СУЩ (контент газеты — предмет design-фазы, brief-контракт
  достаточен); передать design-фазе явным входом вместе с F1=газета+canary из backlog.
- Порядок «выбор ветки стен на анкете → спайк стен ДО BUILD»: если спайк провалит
  выбранную ветку, потребуется возврат на гейт — цена петли гейт→спайк→гейт не
  названа; дешёвая страховка — спайкнуть ОБЕ ветки (deny-поверх-bypass И auto+sandbox)
  за один заход.
