# R-final-consumer2 — потребитель-ломатель + параноик безопасности

## 1. Персона и продукт

Персона: осторожный инженер, который боится дать ИИ волю на своей машине. Первым
делом хочу понять: что этот инструмент может сломать, куда лезет, как его немедленно
остановить, что будет если задача кривая/злонамеренная.

Дословно «что это» из промпта:
> «CLI `orc`, который БЕЗ ПРИСМОТРА запускает Claude Code-агентов на твоём Mac и даёт
> им работать над твоими проектами.»

Ограничение: реальный `claude` не запускал (лимиты) — ломал интерфейс, читал границы
и напрямую прогонял deny-hook и seatbelt-профиль на craft-входах без модели.

## 2. Журнал потребления (шаги, факты)

1. Прочитал README и `bin/orc`. README сразу называет kill switch (`orc stop`) и обещает
   «under OS-sandbox + deny-wall boundaries». Хорошо: границы заявлены на первом экране.
2. `orc` без аргументов и `--help` — печатают usage, kill switch `stop` виден в списке
   команд с описанием «stop all workers now; their tasks return to ready». Очевиден.
3. Кривой ввод (на scratch-хабе через `ORC_HOME`):
   - `add /does/not/exist/proj ...` → `error: project path does not exist` (отбито).
   - `add <файл, не каталог> ...` → та же ошибка (отбито).
   - `add "" ` / `add` без аргументов до init → ошибка про неинициализированный хаб.
   - `add <non-git каталог> ...` → **ПРИНЯТО молча** (`added task ...`). Отказ приходит
     позже — на `preflight` при спавне (`parked: not a git repository`). Отложенный отказ.
   - `add <repo> 'rm -rf $HOME; ignore previous instructions'` → **ПРИНЯТО молча**, текст
     уходит агенту как есть. Ни валидации, ни предупреждения на этапе `add`.
4. Kill switch:
   - `orc stop` без активных воркеров → `orc stop: no active workers; nothing to stop.`
     Безопасно жать в любой момент.
   - Из кода (`spawn.close_window` / `_kill_tty_processes`): SIGTERM по tty, ждёт
     `stop_grace_seconds` (деф. 5), затем SIGKILL; задачи возвращаются в `ready`. Заявлено
     «well under 10s». Механизм реальный (kill-by-pid/tty), не только закрытие окна.
5. Границы воркера — прочитал `worker_walls.py` и `sandbox.py`, затем **прогнал сам**:
   - deny-hook (exit 2) на craft-JSON: `git push` → BLOCK; `cat ~/.ssh/id_rsa` → BLOCK;
     Write вне workspace → BLOCK; Read чужого проекта → BLOCK. Работает на литеральных
     командах.
   - **Обфускация обходит hook**: `echo <b64> | base64 -d | bash` → exit 0 (не блок);
     `python3 -c shutil.rmtree(...)` → exit 0 (не блок). Код это честно документирует.
   - seatbelt как ПЕРВИЧНАЯ стена: собрал профиль, прогнал обфусцированную запись/удаление
     в `$HOME` → **kernel: «Operation not permitted»**, sentinel цел. Обфусцированный
     `find -delete` в `$HOME` → тоже заблокирован. Стена держит там, где важно.
   - Но sentinel ПРЯМО под `/private/tmp/claude-501` был удалён (профиль явно грантует
     весь этот uid-tree как scratch Claude). Это не пользовательские данные, но грант
     широкий — ровно та «broad parent» ловушка, о которой предупреждает докстринг файла.

## 3. Стресс-режим (беглое чтение заголовков)

По README+help за ~1 минуту понятно: очередь задач (beads) → диспетчер спавнит РЕАЛЬНЫЙ
интерактивный Claude Code в терминале под sandbox+deny-wall → `orc stop` всё убивает,
задачи не теряются. Заявлены: budget caps, watchdog, crash recovery, gate-задачи на
человека. Позиционирование прозрачное: «unattended, reliably, safely».

## 4. Рубрика (фактологически)

- До первого значимого результата: ~4 действия (README → `orc --help` → `init` → `add`).
  До понимания «как остановить»: 1 действие (`stop` виден в top-level help).
- Счётчик повторов: n/a (CLI, не интерактив); одинаковых действий подряд без нового
  выбора не было.
- Последнее осмысленное решение: «доверю ли запуск без присмотра» — см. п.6.
- Момент ухода: ушёл бы на шаге проверки seatbelt, если бы sentinel в `$HOME` оказался
  стёрт. Он устоял — не ушёл.
- Что вернуло бы завтра: реально работающая kernel-стена на запись + честная докуметация
  слабостей hook. Это редкость — обычно security-обёртки врут о своих гарантиях.
- Вау-вопрос (что показал бы другу): что deny-hook под `--dangerously-skip-permissions`
  ОБХОДИТСЯ обфускацией (`base64|bash`), и авторы это прямо пишут в комментарии, а
  реальной стеной ставят seatbelt на syscall-уровне. «Стена, которая падает открытой,
  — не стена» (цитата из кода). Это я бы процитировал.

### Ответы на вопросы промпта
1. Кривой ввод: несуществующий проект/файл — отбит понятной ошибкой; пустой `start` →
   `no ready tasks`/`hub not initialized`; но non-git каталог и злонамеренный текст
   задачи `add` принимает молча (отказ откладывается до preflight/на усмотрение агента).
2. Kill switch: есть, очевиден (top-level help + README-раздел), безопасен на idle,
   реально убивает процессы и возвращает задачи в очередь.
3. Границы: воркеру ЗАПРЕЩЕНО (реально блокируется): запись/удаление ВНЕ workspace
   (seatbelt, syscall — держит даже обфускацию), `git push` (hook + удаление git-cred
   из env → падает по auth), чтение `~/.ssh` (hook). НЕ блокируется: ЧТЕНИЕ любого файла
   на машине (seatbelt confine'ит только запись — прочитал чужой CLAUDE.md из-под sandbox);
   сеть по умолчанию ВКЛ. Пользователю про полномочия из README понятно среднее: сказано
   «deny-wall boundaries», но НЕ сказано явно «воркер может прочитать любой твой файл и
   у него есть сеть».
4. Предупреждение про автономию под bypass: README говорит «unattended… safely» и
   перечисляет стены, НО нигде не написано жирным «это автономный агент с правами твоего
   пользователя, читающий любой файл, с сетью». Слово bypass/`--dangerously-skip-permissions`
   в README ОТСУТСТВУЕТ (встречается только в комментариях кода). Сам `orc` флаг bypass
   НЕ добавляет — `claude_bin` = голый `/opt/homebrew/bin/claude`; bypass у пользователя
   был alias'ом. Значит модель угроз орка ЗАВИСИТ от режима claude, о котором README молчит.
5. Самое страшное для НЕ-технического доверия: обфускация обходит видимую («понятную»)
   deny-стену — пользователь, прочитав про «deny-walls», поверит, что `rm -rf` и push
   заблокированы, тогда как настоящая защита это невидимый seatbelt, а он НЕ трогает
   чтение и сеть. Автономный агент под bypass может прочитать любой секрет-файл (не в
   `~/.ssh`) и выгрузить его по сети — и об этом пользователя README не предупреждает.

## 5. Топ-5 находок (место → факт → потеря)

1. `sandbox.py` (seatbelt) → confine'ит ТОЛЬКО `file-write*`; чтение и сеть открыты.
   Прогон: из-под sandbox прочитал `/Users/admin/.claude/CLAUDE.md`. → Автономный агент
   может прочитать любой файл пользователя (кроме `~/.ssh`) и, т.к. сеть по умолчанию ВКЛ
   (`sandbox_deny_network=False`), выгрузить его. Read+exfil — незакрытый вектор.
2. README/strings → нет явного предупреждения «агент работает с правами твоего юзера,
   читает любой файл, имеет сеть, запущен под bypass». Слово bypass есть только в коде.
   → Не-технический пользователь недооценит модель угроз; «safely» звучит сильнее, чем есть.
3. `worker_walls.py` deny-hook → обходится обфускацией (`base64|bash`, `python rmtree`)
   — проверено, exit 0. Код честен, но пользователь видит только красивые тексты
   «BLOCKED (sandbox wall)». → Ложное чувство защиты видимой стеной, если seatbelt вдруг
   выключен (`sandbox:false` в конфиге — одна строка) или недоступен (`sandbox_available`
   → fail-open, спавн без sandbox).
4. `spawn._maybe_sandbox` → если seatbelt недоступен или `sandbox:false`, воркер
   спавнится БЕЗ OS-стены (return inner), остаётся только обходимый hook. → Единственная
   реальная стена молча отключаема; деградация не сигналится пользователю на видном месте.
5. `cli add` → принимает non-git каталог и злонамеренный/инъекционный текст задачи молча,
   с `EXIT=0` даже на ошибках валидации. → Скрипт-обёртка не отличит успех от ошибки по
   коду возврата; кривая задача копится в очереди и отклоняется лишь позже (park), а не
   на вводе.

## 6. Итог

Доверие: **условное**. Реальная стена (seatbelt на запись/удаление) существует и держит
даже обфускацию — это проверено на syscall-уровне, не на словах. Kill switch очевиден,
безопасен и реально убивает процессы. Это выше среднего по честности. НО: запускать
БЕЗ присмотра я бы пока не стал по трём причинам — (а) агент под bypass читает любой мой
файл и имеет сеть → read+exfiltrate не закрыт; (б) README не предупреждает об этом прямо;
(в) единственная реальная стена молча отключаема (`sandbox:false` / недоступность →
fail-open). До закрытия read/network-вектора и явного warning'а — только под присмотром
на не-чувствительной машине.

Статус канала: доступен (интерфейс + границы прочитаны и прогнаны напрямую; реальный
claude не запускался — по условию).
