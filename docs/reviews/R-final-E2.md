## 1. Паспорт вкуса — дисциплина CLI-вывода (реальный прогон)

Правила паспорта (brief.md:63): plain text; важное первой строкой; ≤80 колонок;
без эмодзи кроме статус-символа ✓/✗/⏸; без «Successfully!»; каждая команда имеет `--json`.

### Что ПРОШЛО
- **Газета сводкой ПЕРВОЙ строкой** — подтверждено:
  `смена: 1 готово, 0 ждут тебя, 0 упало; съедено 46% окна` (строка 1), заголовок ниже.
  Прошлый мел M1/F14 «сводка на 2-й строке» — ЗАКРЫТ (report.py:119 summary_line первым).
- **plain text, grep-able** — да; статус-глифы ✓ ✗ ⏸ ◐ (⏸ на гейт-карточке — разрешён).
- **Ширина живого status/newspaper (без длинных путей)** — max 55 символов, под 80.
  (Мерил в СИМВОЛАХ, не байтах: `awk length` даёт 90 из-за 2-байтной кириллицы UTF-8 — ложная тревога.)
- **Без «Successfully!»/восклицаний** — grep пуст.
- **`--json` валиден у** init / add / status / status --newspaper / stop / setup / install —
  все прошли `python3 -m json.tool`. Идемпотентность init видна в JSON (`created:true/false`).
- **DONE / DONE-WAVE / BETA различимы** (прошлый мел F14 «схлопываются в готово» — ЗАКРЫТ):
  `✓ готово` / `✓ готово (предложена волна)` / `◐ бета — ждёт твоего решения` (report.py:169, strings.py:129-131).
- **Терминологический разнобой .beads/~/.orc** (жалоба consumer R-M1) — ЗАКРЫТ:
  и ошибка, и init-сообщение теперь единообразно зовут хаб `~/.orc`, а `.beads/` — очередью внутри
  (strings.py:45-51; живой прогон обеих строк совпал).

### Что НАРУШЕНО (дисциплина вывода)
- **[СУЩЕСТВЕННОЕ] `start --json` НЕ выдаёт цельный JSON.** cli.py:136/144/152/159/173
  безусловно печатают человекочитаемый canary-report + статус-строки в **stdout**, и лишь
  ПОСЛЕ них — JSON-строку. Живой прогон:
  ```
  === canary preflight ===
  [ ok ] bd: ...
  ...
  {"canary_ok": true, "spawned": []}
  ```
  → `orc start --json | jq` захлебнётся преамбулой. Нарушен паспорт «каждая команда имеет
  `--json`» и референс-эталон bd («--json у всего, вывод grep-able»). Фикс: в `--json`-режиме
  человеко-текст → stderr (или подавить), stdout = только JSON. `start` — P0-команда.
- **[МЕЛОЧЬ] Гейт-карточка: строка `ТЗ: <путь>` пробивает ≤80 колонок.** Живой рендер = 121 символ
  (даже для нормального `/Users/admin/projects/<proj>/docs/tasks/<slug>/brief.md` легко >80).
  Нет переноса/усечения. report.py:_gate_card / strings.py RU_GATE_CARD. Signature-фича бьёт бюджет ширины.
- **[МЕЛОЧЬ] Эмодзи вне allowlist.** `⚠` (U+26A0 WARNING SIGN) в строке необратимого гейта
  (strings.py:149) и `•` (U+2022) в очереди (strings.py:124). Паспорт: «без эмодзи, допустим
  статус-символ ✓/✗/⏸». `⚠` — пиктограф, не статус-символ. Низкая тяжесть (несёт сигнал), но формально нарушает.
- **[МЕЛОЧЬ] `daemon` без `--json`.** Паспорт «КАЖДАЯ команда имеет --json»; `daemon` — цикл, не запрос,
  оправдательное исключение, но формально по букве не соответствует.

**Паспорт вкуса: FAIL** — доказательство: `start --json` печатает не-JSON преамбулу в stdout
(живой вывод выше); гейт-карточка `ТЗ:` = 121 кол. > 80 (живой рендер). Signature-элемент
«утренняя газета» существует, заметен и функционален (сводка первой строкой, гейт-карточки
скоуп/планка/полномочия/цена, DONE/BETA/WAVE различимы) — но дисциплина вывода нарушена в двух
местах, одно из них (start --json) — на P0-команде.

## 2. Detail pass кода (src/orc/*.py — перечислил и сверил)

- Заглушек «под готовое» / TODO/FIXME/NotImplemented — НЕ найдено (grep чист; единственный «stub»
  в комментарии spawn_ghostty.py:131 — про SIGKILL, не заглушка). Все `pass` — легитимный swallow except.
- Обработка ошибок: `orc add /nonexistent` → чёткая ошибка + exit 1 (проверено). `add` до `init`
  → ясная подсказка + exit 1. Пустая очередь → `no ready tasks; nothing to start`, exit 0.
- Гейт-карточка тянет scope/bar/authority/cost/irreversible из bd-метаданных (report.py:_gate_card) — работает на живом hub.
- **[СУЩЕСТВЕННОЕ] newspaper() не защищён от BeadsError.** report.py:110 → `_gate_card` (стр.192)
  вызывает `beads.show(hub, ...)`, который бросает `BeadsError`; вокруг гейт-карточек в newspaper()
  НЕТ try/except, cmd_status (cli.py:223) тоже без обёртки. Если bd транзитно недоступен ПРИ
  запаркованном гейте → вся газета падает вместо деградации. Signature-фича обязана деградировать,
  не крашиться (доказано синтетически: `newspaper(state с parked, bad_hub)` → `BeadsError: bd not found`).
- **[МЕЛОЧЬ] `_git_push_in` мимо `env VAR=val git push`.** worker_walls.py:134. Ток-скиппер снимает
  VAR= ДО цикла env/sudo, но не ПОСЛЕ → `env FOO=1 git push` и `sudo env A=1 git push` дают
  предикат False (проверено: `_git_push_in('env FOO=1 git push')==False`). Это слой-1 (читаемая причина),
  который дизайн сам объявляет обходимым; настоящая стена (env-prefix) держит (см. §3). Но дизайн
  заявляет «F1 блокирует литерал git push» — литерал с env-префиксом не ловится.
- Согласованность сообщений init/хаб — единообразна (см. §1, разнобой закрыт).
- `stripped_env` покрыт unit-тестом изолированно (test_worker_walls.py:165), но НЕ применяется на спавне (см. §3).
- pytest: **210 passed** (44.9с), 0 fail.

## 3. Полнота границ песочницы (мой центральный чеклист)

Требование brief.md:113-119: deny-стены (git push, rm -rf вне workspace, чтение ~/.ssh и чужих
проектов, запись вне workspace); MCP-allowlist дефолт пуст; env чистится от секретов; bypass+стены;
воркер не может отключить свои стены.

### Что ДОКАЗАНО живьём (сильно)
- **Seatbelt = основная стена записи РЕАЛЬНА.** Живой `sandbox-exec -f <profile>`:
  - запись внутрь workspace → exit 0;
  - запись в `$HOME/sentinel` → `Operation not permitted`, sentinel НЕ создан;
  - запись в `~/.claude/` root → `Operation not permitted` (воркер НЕ может писать свои enforcement-файлы);
  - обфусцированное удаление вне workspace (`python shutil.rmtree($HOME/victim)`) → victim ВЫЖИЛ.
  Профиль корректно узкий: `(subpath workspace)` + narrow claude-runtime subdirs; skills/agents/settings.json
  НЕ writable (проверено грепом профиля + живой записью).
- **Хук-стены (PreToolUse exit 2) ловят наивные пути** — прогнал 14 payload-ов (в файлы, чтобы не
  триггерить глобальный pipeline-хук): git push (plain/cd/`-C`), cat ~/.ssh/id_rsa, Read ~/.ssh,
  Read/Write чужого проекта, rm -rf вне ws — ВСЕ exit 2. rm внутри ws / write внутри / `git commit -m "push"` — exit 0. 13/14 корректно.
- **git-push CAPABILITY снят (реальная стена).** Живой прогон под `push_neutralizing_export_prefix()`:
  даже `env FOO=1 git push origin HEAD` → `unable to read askpass '/usr/bin/false'` →
  `could not read Username: terminal prompts disabled`, exit 128, НИЧЕГО не запушено. Стена держит
  ВНЕ зависимости от обфускации хука.
- **MCP-allowlist дефолт ПУСТ** — `generate_worker_settings('/tmp/ws')` →
  `enabledMcpjsonServers: []`, `enableAllProjectMcpServers: False`. Подтверждено.
- **permissions.deny** = `Bash(git push:*)`, `Read(~/.ssh/**)` (defense-in-depth в не-bypass режимах).

### ГЭПЫ границ (реальные)
- **[СУЩЕСТВЕННОЕ] Секрет-env НЕ чистится на спавне.** brief.md:116 «Env воркеров чистится от
  секрет-переменных по denylist… прод-креды недоступны ПО ПОСТРОЕНИЮ». Факт: `stripped_env()`
  вызывается ТОЛЬКО в worker_walls._main (CLI `gen`, стр.445) — для ПЕЧАТИ счётчика удалённых.
  `build_start_command` (spawn.py:107) префиксует ТОЛЬКО push-нейтрализацию + ORC_SESSION; НЕТ
  `unset`/`env -u` для ANTHROPIC_API_KEY/AWS_*/GITHUB_TOKEN/*_SECRET и пр. (grep «unset» в spawn/launchagent — пусто).
  Ни один тест не проверяет, что СПАВНЕННАЯ команда чистит секреты (test_env_strip_removes_secrets —
  функция в вакууме). Гарантия «по построению» держится ТОЛЬКО для git-push кредов, не для общего denylist.
  (Смягчение: Terminal.app `do script` стартует свежий login-shell — на практике может не унаследовать
  env демона; но заявленный механизм denylist — мёртвый код на пути спавна, гарантия недоказуема.)
- **[СУЩЕСТВЕННОЕ] Чтение ~/.ssh НЕ walled на syscall-уровне — обфускация уходит.** Seatbelt = `allow
  default` для чтения (греп профиля: НЕТ `file-read*` deny). Хук ловит только известные reader-токены
  в Bash-строке + Read-tool. Живой прогон под seatbelt: `python3 -c "print(open('.../.ssh/id_rsa').read())"`
  → **SENTINEL напечатан** (чтение НЕ заблокировано). Под bypassPermissions воркер эксфильтрует ~/.ssh
  через `python open()` мимо ОБЕИХ стен. brief.md:114 требует стену на «чтение ~/.ssh»; дизайн сам
  цитирует METR «промпт-запреты бесполезны». Read-граница только паттерновая, не syscall. Фикс:
  добавить `(deny file-read* (subpath "~/.ssh"))` в seatbelt-профиль.
- **[МЕЛОЧЬ] `env VAR=val git push`** мимо хук-предиката (см. §2) — но capability-стена держит, так что
  фактического пуша нет; тяжесть снижена до мелочи.

Итог границ: **запись/удаление — стена реальна и держит под обфускацией. Чтение секретов
(~/.ssh) и очистка секрет-env — НЕ полны**: два заявленных бордюра (ssh-read wall, env-denylist)
не enforced на настоящем уровне под bypass+обфускацию. Границы НЕ полны.

## ЧТО ПЛОХО (обязательно)
1. **Опыт оператора: `start --json` не машинно-потребляем** — самая частая утренняя команда старта
   не пайпится в jq, вопреки всему референсу bd-CLI. Отделяет от эталона: bd даёт цельный JSON у всего.
2. **Signature-фича хрупкая**: газета падает по BeadsError при запаркованном гейте (нет try/except),
   и её главная карточка бьёт ≤80 кол. на строке пути. Утренний артефакт обязан деградировать, не крашиться.
3. **Граница «по построению» переоценена**: env-denylist и ssh-read стены заявлены, но на реальном
   пути (спавн под bypass + обфускация) не enforced — есть только их изолированные функции/unit-тесты.
   Разрыв между «функция существует и покрыта тестом» и «граница держит на воркере».

## СОМНЕНИЯ
- Terminal.app `do script` может не наследовать env демона (свежий login-shell), что случайно
  прикрывает env-denylist-гэп на практике — но это не «по построению», а везение среды; под LaunchAgent
  с EnvironmentVariables поведение иное. Заявленный механизм всё равно не работает — это не сомнение, а факт.
- `daemon` без `--json` — оправданно (цикл), не поднимаю выше мелочи.
- Ширину `ТЗ:`-пути ≤80 в CLI трудно держать без усечения; допустимо усечение середины пути с … — но
  сейчас усечения НЕТ вообще.

## РЕЗЮМЕ
Ядро крепкое: seatbelt-стена записи/удаления реальна и держит под обфускацией (сильнейшее звено),
git-push capability снят доказанно, MCP-allowlist пуст, 210 тестов зелены, прошлые мелочи (сводка
первой строкой, DONE/BETA, .beads/~/.orc) закрыты. Но по моему узкому чеклисту два СУЩЕСТВЕННЫХ
разрыва границ (env-denylist не применяется на спавне; ~/.ssh read обходится обфускацией мимо обеих
стен) + СУЩЕСТВЕННЫЙ дефект дисциплины вывода (`start --json` не цельный JSON) + хрупкость газеты
(BeadsError крашит) держат паспорт вкуса на FAIL и границы — неполными.
