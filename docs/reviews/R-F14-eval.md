# R-F14-eval — Внешний скептичный разбор фичи F14 «Замыкание петли»

Класс: software / CLI (детерминированный предикат бьёт мнение). Раунд 1 (первый eval F14).
Метод: без живого claude (окно лимитов почти исчерпано) — seam ORC_SPAWN_CMD_OVERRIDE
делает in-tab программу детерминированной; спавн/окно/tty/kill — реальны. Все прогоны —
мои собственные, не чтение builder-логов на веру.

## Что заявлено (features.md F14, self-pass)
1. consumer-сценарий: задача выполнилась → `orc status --newspaper` показывает «1 готово»
   БЕЗ ручного ls/cat (петля замкнута).
2. вкладка воркера закрывается после терминального статуса (реализовано как СТОП воркера
   по tty; удаление husk-окна best-effort); tab_id в shift.json (не None).
3. `orc status` при непустой ready-очереди до start показывает задачи; `init --help`
   непустой; сообщения о хабе единообразны.
- Доказательство: e2e-loop-close PASS exit 0; 83 passed (68 M1 + 15 F14).

### Г4. Не хардкод? (negative control — мой собственный)
Инструментировал poll_completions: STATE.md = «Status: in progress» → transitions=[],
done=[], воркер остаётся active (len workers=1). Флип STATE→DONE тем же кодом →
transitions=[('t-1','done')]. Вывод: «1 готово» причинно вызвано содержимым STATE.md,
а НЕ безусловным хардкодом. Газета догоняет DONE именно потому, что детектор увидел DONE.

### Г5. Seam ORC_SPAWN_CMD_OVERRIDE — честен?
`build_start_command` при выставленном override возвращает `cd <proj> && <override>` и
claude_bin/prompt в строку НЕ подставляются (проверено: 'claude' not in cmd, prompt not
in cmd). Значит живой claude не спавнится — соответствует ресурс-ограничению.
Ключевое: seam подменяет ТОЛЬКО in-tab программу. Логика замыкания петли
(read STATE.md → detect_terminal_status → bd close → mark_done → close_window/kill-tty)
живёт в dispatcher.poll_completions и исполняется ИДЕНТИЧНО, держит ли вкладка claude
или sleep. Seam НЕ маскирует завершение: он вообще не касается пути детекции. Спавн окна,
резолв tty (`window_tty`), SIGTERM процессов на tty (`_kill_tty_processes`), close(window)
— все реальны и выполнены в моём прогоне. Seam честен.

### Г6. pid-None пофикшен?
`spawn_one` сохраняет `tab_id = detail` (window id spawn_terminal) в shift.json; session =
"window id N". Мой прогон spawn_one с mock spawn_terminal→'7777': shift.json.tab_id='7777',
user-line = «spawned worker … (Terminal window id 7777)» — bare None НЕТ.
E2E: реальный tab_id 4872 в shift.json. Consumer-симптом «(pid None)» устранён: воркер
трекается и убивается по window-id→tty. NB: внутреннее поле `pid` МОЖЕТ остаться None,
если lsof не нашёл cwd-совпадение (best-effort), но оно нигде не рендерится пользователю,
а reconcile имеет cwd-fallback (`worker_pids`) для восстановления живого PID. Симптом,
на который жаловался consumer (видимый «(pid None)»), закрыт.

### Г7. Consumer-дыры #2/#3/#5 (полировка)
- #2 (задача невидима до start): `live_status` при непустой ready и не-стартованной смене
  печатает секцию «в очереди» (queued_lines) — код есть, путь реальный. E2E косвенно
  (add до start не проверяет вывод очереди явно, но unit + код подтверждают).
- init --help: `sub.add_parser("init", ...)` имеет непустой help + description про
  глобальный ~/.orc и «один beads на все проекты». Терминологический разнобой .beads/.orc
  снят: init-строка = «orc hub initialized at … (beads queue ready inside .beads/)».
- #5 (смешанный язык): по features.md — ЯВНОЕ решение пользователя, не баг. Не блокер.

паспортом промпта.

## ЧТО ПЛОХО
- Опыт/петля: «две дороги» неполны для целевой планки. Seam-детерминизм — реальная дорога,
  но ЕДИНСТВЕННЫЙ живой прогон (~22с) НЕ зафиксирован в docs/evidence/F14/ — там только
  seam-лог (window 4867) и unit-лог. Заявка features.md о «~22с живьём» в evidence
  ОТСУТСТВУЕТ. Для CLI-петли это приемлемо (seam покрывает механизм детекции целиком,
  claude нужен лишь чтобы СОздать STATE.md, что seam честно эмулирует), но заявленной
  «дороги живого прогона» в артефактах я не нашёл — только заявление. Это слабость
  доказательной базы, не механизма.
- Порог таймаута: E2E ждёт до 300с, но реальный claude пишет STATE.md не за 3с sleep, а
  минуты; на исчерпанном окне (в моём прогоне «1 min remaining», «100% окна») канарейка
  всё равно пропустила старт — canary проверяет наличие окна, но не достаточность. Для
  реальной смены на почти-пустом окне воркер стартует и упрётся в лимит; F14 это не
  ловит (это скоуп F5 admission, todo). Петля замкнётся по gate/fail, но опыт «вернулся
  за кофе» на пустом окне будет «0 готово» правомерно.
- Газета не различает «done по DONE» и «done по BETA» — обе схлопываются в «готово».
  Для консьюмера BETA (частичный релиз) и DONE — разный сигнал забирать/не забирать;
  сейчас неразличимы. Мелочь опыта, но реальная.

## СОМНЕНИЯ
- Seam делает воркер «sleep 600 после DONE», т.е. процесс на tty жив — это правильно
  моделирует lingering claude, и kill-by-tty реально его валит. Но реальный claude может
  держать НЕСКОЛЬКО процессов/дочерних; `_kill_tty_processes` бьёт всех на tty — вероятно
  ок, но на живом claude не проверено в этом раунде (окно лимитов). Подозрение слабое:
  ps -t ловит всю сессию tty.
- `detect_terminal_status`: `_GATE_RE.search(state_text)` по ВСЕМУ тексту — если в
  STATE.md ранней фазы упомянуто слово «ждёт ответа» в другом контексте, задача ложно
  запаркуется. Узкий риск,但 регэксп широкий. Не блокер (gate безопаснее, чем ложный close).
