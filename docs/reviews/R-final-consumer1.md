# Consumer review — orc CLI (свежий потребитель, финальный проход)

## 1. Персона и продукт

- **Персона**: разработчик, утро понедельника. Хочу накидать 3-4 задачи по двум
  проектам, уйти, вернуться и за 30 секунд из ОДНОЙ команды понять: что готово, что
  ждёт меня, что упало. Документацию не читаю.
- **«Что это» (дословно из промпта)**: «CLI `orc`, который сам прогоняет твои задачи
  через Claude Code и утром показывает, что сделано.»
- Реальную смену с живым claude НЕ запускал (берегу лимиты). Прожил интерфейс:
  `orc`, `--help`, `add`, `status`, `status --newspaper`, плюс проверил F14-догон и
  газету через seam на изолированном `ORC_HOME`.

## 2. Журнал потребления (шаги, факты)

1. `./bin/orc` без арг → выдаёт usage (тот же, что `--help`), но **exit 1**. `--help`
   → тот же текст, exit 0. Понял список подкоманд за 1 действие.
2. `orc add "fix the login bug in project-a"` (как естественно печатает человек —
   текст первым) → **ошибка**: `error: project path does not exist: fix the login
   bug in project-a`. Споткнулся: первый позиционный арг это ПУТЬ ПРОЕКТА, а не текст.
   Ошибка винит «project path», но не подсказывает форму `orc add <project> <text>`.
3. `orc add --help` → увидел `[project] [text]`. Понял порядок. ~1 доп. действие.
4. `orc init --help` → длинный внятный текст: хаб один на все проекты в `~/.orc`,
   идемпотентен. init-непонятность (жалоба M1) снята: описание чёткое.
5. Хаб `~/.orc` уже существовал (был прогон в 06:03). `orc init` → «already
   initialized». `add` НЕ жаловался на отсутствие хаба — жаловался на путь. ОК.
6. Завёл 4 задачи против реальных папок (2 проекта, 1 urgent -p0, 1 --gate со
   scope/bar/authority/cost). Все 4 → `added task orc-xxx -> <path>`, exit 0.
7. `orc status` (живой) на дефолтном хабе → показал ТОЛЬКО пул-строку
   `пул: 46% окна, 161 мин осталось, RAM ok`. **Моих 4 задач НЕ видно.**
8. `orc status --newspaper` → `1 готово, 0 ждут, 0 упало` + `✓ orc-xxa готово` —
   это ЛЕВАЯ задача из прошлого прогона, не мои. Мои 4 нигде.
9. `bd ready` (сырьё) → показывает ВСЕ 4 мои задачи. Значит очередь их знает, а
   `orc status` их прячет.
10. Прочитал `report.py`: превью очереди («── в очереди ──») показывается ТОЛЬКО при
    `not started AND not workers AND not parked`. Дефолтный `~/.orc/shift.json` имеет
    залипшее `"started":"…06:03"` + одну done-задачу → orc думает, что смена уже
    идёт/завершена, и превью очереди пропускает.
11. Проверил на ЧИСТОМ хабе (`ORC_HOME=…/orc-home-test`): те же 4 задачи → `orc status`
    показал «── в очереди (смена не запущена, `orc start`) ──» со списком id + путей +
    инструкцией запустить `orc start`. **Тут M1-фикс работает.**
12. F14-догон: собрал shift.json с worker'ом «в работе», чей STATE.md на диске = DONE.
    `orc status --newspaper` → перекинул задачу в «── завершено ── ✓ готово», сводка
    `1 готово`. **Догоняет DONE — M1-жалоба снята.**
13. Богатая газета (1 done / 1 gate / 1 failed): сводка на строке 1, секции «ждут
    твоего решения» (полная карта решения: скоуп/планка/полномочия/цена/ТЗ-путь),
    «завершено» (✓), и ✗ упало. См. находки #2.

## 3. Стресс-режим (беглый взгляд на газету)

Строка 1 газеты — `смена: N готово, N ждут тебя, N упало; съедено N% окна` —
grep-абельна и за ~5 секунд отвечает на все три вопроса персоны + бюджет. Это сильная
сторона: 30-секундный глазок реально закрывается ОДНОЙ строкой. Заголовок «СМЕНА orc»
идёт ПОСЛЕ сводки (F14: сводка вынесена на строку 1 — подтверждаю).

## 4. Стандартная рубрика (факты)

- **Действий до первого значимого результата**: постановка задачи — 3 действия
  (наткнулся на ошибку порядка арг → `add --help` → корректный `add`). Чтение статуса
  — 1 команда, НО на дефолтном хабе она дала неверную картину (см. #1 находок); на
  чистом хабе — верную с 1-й попытки.
- **Счётчик повторов**: макс. серия одинаковых действий без новой информации — 4
  (четыре `orc add` подряд; каждый даёт новый id, так что информация новая — по сути
  повторов-впустую нет).
- **Последнее осмысленное решение**: пометить одну задачу как `--gate` со scope/bar/
  cost — это меняет то, что worker НЕ примет решение сам, а припаркует его мне в
  «ждут твоего решения» с картой. Решение отражается в газете корректно.
- **Момент ухода**: на шаге 7-8. Если бы я НЕ полез в исходники, я бы на дефолтном
  хабе увидел «1 готово, 0 ждут» и решил, что мои 4 задачи потерялись → закрыл бы
  терминал в растерянности. Спасло только то, что я программно проверил `bd ready`.
- **Что вернёт завтра**: газета-строка-1 и карта gate-решения — ради них вернусь:
  один `orc status --newspaper` реально заменяет чтение логов.
- **Вау-вопрос (что показал бы другу / процитировал)**: одну строку газеты —
  `смена: 2 готово, 1 ждут тебя, 0 упало; съедено 43% окна`. Это и есть продукт в
  одной строке: «поставил задачи вечером, утром одна строка сказала всё». Показал бы
  именно её.

## 5. Топ-5 находок

1. **`orc status` на «грязном» дефолтном хабе прячет только что добавленные задачи.**
   Место: `orc status` при залипшем `~/.orc/shift.json` (`started` + старый done).
   Факт: 4 добавленные задачи невидимы; вместо них показана левая «1 готово» из
   прошлого прогона; `bd ready` при этом их видит. Превью «── в очереди ──»
   показывается лишь при `not started`. Почему потеря: ровно сценарий персоны (утром
   добавил, глянул статус) даёт ложное «задачи пропали»; это M1-жалоба №2 в новом
   обличье — фикс есть, но обходится залипшим shift.json прошлой смены. Нет команды
   «сбросить/начать новую смену», после которой очередь снова видна без запуска
   реального claude.

2. **Упавшие задачи не имеют своего заголовка — сидят под «── завершено ──».**
   Место: `report.py` newspaper + `strings.py` (есть `RU_SECTION_GATES/RUNNING/DONE`,
   нет `RU_SECTION_FAILED`). Факт: `✗ … упало:` рендерится сразу после done-блока без
   отдельной шапки. Почему потеря: персона явно ищет «что упало»; беглый глаз может
   прочесть провал внутри «завершено». Число в сводке отдельное, но детальный блок
   сливает провал с успехом.

3. **`orc add "<текст>"` (текст первым) падает с обвиняющей путь ошибкой.**
   Место: `orc add`, первый позиционный = путь проекта. Факт: `orc add "fix the bug"`
   → `error: project path does not exist: fix the bug`; нет подсказки формы
   `orc add <project> <text>`. Почему потеря: самый естественный первый ввод нового
   пользователя ломается, а сообщение уводит не туда.

4. **Токены в газете показывают глобальный счётчик, не пер-задачу (до ~81M «ток.»).**
   Место: `✓ … готово ~81219995 ток.` в моём F14-тесте. Факт: значение — глобальный
   ccusage-каунтер, а не спенд задачи (README/F12 честно помечают ccusage-лаг). Почему
   потеря: разработчик, следящий за лимитами, увидит «81 млн токенов на задачу» и
   запаникует; сводка `съедено N% окна` надёжнее — цифра «ток.» рядом её подрывает.

5. **Живой `orc status` не перечисляет ready-задачи, когда смена ИДЁТ.**
   Место: `live_status` — при running-смене показывает gates + running + пул, но НЕ
   очередь «что ещё ждёт запуска». Факт: во время смены нельзя одной командой увидеть,
   сколько задач ещё в очереди. Почему потеря: персона хочет «что ждёт тебя» — а «ждёт»
   тут = только gate-парковки, не оставшийся бэклог смены.

## 6. Стало ли лучше vs M1

- **Газета догоняет DONE (M1 №1)**: ДА, исправлено — подтверждено на seam-тесте (шаг 12).
- **init непонятен (M1 №3)**: ДА, лучше — `orc init --help` даёт внятное описание хаба.
- **Задача, добавленная до `start`, невидима (M1 №2)**: ЧАСТИЧНО — на чистом хабе
  превью очереди работает; но залипший shift.json прошлой смены снова прячет очередь
  (находка #1). Фикс есть, но не покрывает «вторую смену того же дня».
- **Смешанный язык**: оставлен намеренно — читаем, не мешает (RU-UI + EN-текст задач).

## Итог

`ЧЕКПОЙНТ: ПРОЙДЕН` — основная ценность получена: постановка задач работает, а
газета-строка-1 за ~5 секунд отвечает на «готово/ждёт/упало/бюджет», и F14-догон до
DONE подтверждён. Порогов N/M в промпте нет → вердикт по повторам/механикам не выношу
(сырьё: 4 add подряд, повторов-впустую 0).

Оговорка к «ПРОЙДЕН»: находка #1 — на дефолтном хабе с залипшей прошлой сменой
`orc status` показывает ложную картину и прячет только что добавленные задачи; в
реальном сценарии персоны (вторая смена за день) это едва не привело к «ушёл бы в
растерянности» на шаге 7-8. Ценность спасена только альтернативным путём (чистый хаб /
`bd ready`), не основной командой.

## Вердикт (грамматика review-verdict.md)

ВЕРДИКТ: ПРИНЯТО-С-ЗАМЕЧАНИЯМИ

ЧТО ПЛОХО:
- **Опыт/петля (находка #1):** на «грязном» дефолтном хабе `orc status` прячет только

  worker=в работе, STATE.md на диске=DONE → `status --newspaper` перекинул в «завершено
