## 2. Фичи как ИСХОДЫ; бинарность и исполнимость приёмки

Приёмка F1–F9 — в основном бинарные исходы с исполняемой проверкой (`bash .verify/*`,
`pytest`), evidence/ путь у каждой. Критериев-«действий» («проанализировать») нет —
зачёт по принципу 2. Но:

- **G12 (допуск ≤60 сек) растворился.** F4 приёмка: «допуск: ready≠∅ + окно/RAM ok →
  spawn ≤60 сек» — есть! Но G12 — прокси-гейт субъективного критерия «по умному» (ТЗ,
  §Субъективные критерии). В F4 он смешан с admission-логикой лимитов, без явной
  пометки «это и есть измеритель „по умному“». Не блокер (критерий физически покрыт),
  но связь гейт↔фича невидима — МЕЛОЧЬ (следствие дыры трассируемости §1).
- **F1 приёмка «≤150 слов»** — бинарно и исполняемо (счёт слов первого экрана). Зачёт.
- **F5 «живой Bash-вызов ≥2 мин НЕ убивается (0 ложных kill)»** — исполняемый
  контр-тест, лечит Гудхарт watchdog (убить всё = 0 петель). Зачёт по принципу 4.
- **F7 негативный спайк** — исход бинарный («ЗАБЛОКИРОВАНЫ, вывод в evidence/F7/»).
  Зачёт (см. §7).
- **F9 «0 регрессий»** — характеризационный набор до/после. Зачёт (см. §8).

## 3. Гранулярность (1 такт / ≤3 критерия) — F4/F5 не раздуты?

Дисциплина статусов сама заявляет «1 такт/≤3 критерия». Проверка:
- F1: 4 критерия приёмки. **Превышает ≤3.** Оправдано? F1 — walking skeleton = НАМЕРЕННО
  многотактовый сквозной срез (add→start→canary→spawn→status). Это единственная фича,
  где сквозной такт — суть. Терпимо, но формально нарушает собственное правило → МЕЛОЧЬ
  (пометить: «скелет — исключение, критерии = станции сквозного пути, не отдельные фичи»).
- F4 (Admission+back-pressure): 3 критерия, но «Что» несёт ЧЕТЫРЕ механизма (ccusage-окно,
  free-RAM, лимит-строки, бюджет-кап). Лимит-строки (4 класса: session/weekly/Opus/429) +
  RAM-допуск + бюджет-кап смены — это фактически 3 разных гейта ТЗ (G4, G8, часть G12),
  слитых в одну фичу. Раздута по ОБЪЁМУ ответственности, хоть и 3 критерия. Риск: «частично»
  (лимиты есть, бюджет-кап нет) пройдёт как один self-pass. СУЩ-кандидат → фиксирую как
  СУЩ ниже (бинарность под угрозой: фича с 4 механизмами не бинарна).
- F5 (Watchdog+восстановление+lease): 3 критерия, но «Что» несёт петлю-детект + тишину +
  внешнюю проверку + kill + рестарт-cap + lease-TTL + kill-9-recovery. Это G5 + G6 в одной
  фиче. Аналогично F4 — раздута ответственностью. G6 (восстановление после kill -9
  диспетчера) — отдельный гейт цели (crash-safe), заслуживает своей бинарной приёмки, а не
  третьего булита F5. СУЩ ниже.
- F2/F3/F6/F7/F8/F9: по 3 критерия, один такт каждая. Зачёт.

## 4. F1 walking skeleton содержит SIGNATURE (газета+canary), не заглушку?

**ЗАЧЁТ по принципу 13.** F1 приёмка требует: canary-предполёт печатает отчёт И при
подставном фейле смена не стартует (реальный canary, не заглушка); `orc start` спавнит
РЕАЛЬНЫЙ интерактивный терминал (не headless), задача выполняется, hello.txt появляется
(signature — сквозная автоматизация, самый рискованный элемент — реально в скелете);
`orc status` печатает газету с первой строкой-сводкой ≤150 слов. Проверка
`.verify/e2e-skeleton.sh` ассертит hello.txt + формат газеты. Это ровно лекарство
failures.md 2026-07-05 («walking skeleton»/«signature-в-скелете»): самый ценнообразующий
и рискованный элемент (реальный спавн claude + газета) — в F1, не отложен. Хорошо.

Слабость: canary в F1 проверяет «bd/auth/ccusage/RAM/spawn-терминала», но R-brief р.1
СУЩ-6 требовал именно спавн терминала из контекста LaunchAgent (TCC Automation
спрашивается лениво). F1 гоняется из ИНТЕРАКТИВНОЙ сессии сборки, где TCC уже дан → canary
в F1 может показать зелёный, а из LaunchAgent (F8) — красный. Signature скелета
не воспроизводит боевой контекст запуска. Не блокер для F1 (скелет легитимно упрощает),
но пометить: TCC-из-LaunchAgent — критерий приёмки F8, не считать закрытым в F1 → ЧТО ПЛОХО.

## 5. Золотой путь (F1/F3/F6) выбран верно?

**ЗАЧЁТ.** F1 (скелет+signature — переживание ценности целиком), F3 (диспетчер-ядро —
сердце автономии), F6 (гейт-опыт — единственная точка человека, ось ступени 4). Это
2–4 фичи, критичные для опыта, каждая обоснована. F1 несёт «утреннюю газету» (ставка
вау), F6 несёт гейт-карточку (ставочная фича). Выбор согласован со ставкой вау ТЗ и
тактом опыта. Возражение: F4 (admission/back-pressure) — не в золотом пути, хотя без
детекта лимитов автономная смена «сожжёт weekly-кап за утро» (cut-list R-brief:
«окно — валюта продукта»). Но F4 — надёжность, не переживание ценности; золотой путь =
ценность, страховка отдельно. Выбор корректен; F4 как не-золотой приемлемо.

## 6. ADR: победитель имеет замеры из спайка?

**ADR-0001 (python-stdlib) — ПРОБЛЕМА ДОКАЗАТЕЛЬСТВ.** Таблица заявляет замеры:
«`/usr/bin/python3` = 3.9.6; `import tomllib` → ModuleNotFoundError; `import json` OK;
`/usr/bin/jq` есть». Принцип 7: «ADR, где победивший вариант без артефакта спайка =
БЛОКЕР». Перепроверка корпуса workspace: в `.spikes/probe/` есть terminal.md, ccusage.md,
beads.md, happy.md, heartbeat.md, launchagent.md — но **НЕТ спайк-артефакта для
python/tomllib/jq**. terminal.md (прочитан) фиксирует claude-версию, osascript, Keychain,
CLI-фичи — НО ни строки про `python3 --version`, `import tomllib`, `jq`. Замеры ADR-0001
не имеют прослеживаемого артефакта: единственная опора — сам текст ADR («spike Bash»,
«RS-02/пробы»), а RS-02 (прочитан целиком) — про точки патча pipeline, python-версии там
нет. Утверждение решение-несущее (весь стек висит на «tomllib нет в 3.9.6»). Число без
артефакта = НЕ ДОКАЗАНО.

Смягчение: python 3.9.6 + отсутствие tomllib (tomllib добавлен в 3.11) — правдоподобно
для системного macOS python; jq в /usr/bin — типично. Но правдоподобие ≠ замер, а профиль
критика требует перепроверки, не веры. **БЛОКЕР (доказательство): приложить спайк-артефакт
(вывод `python3 --version`, `python3 -c "import tomllib"`, `which jq`, `python3 -c "import
json"`) в `.spikes/probe/` и сослаться из ADR-0001** — иначе решение стека на непроверяемом
основании (и любой параллельный вывод «на этой машине python новее» его роняет).

**ADR-0002 — то же основание, тот же дефект.** Опирается на «`import tomllib` →
ModuleNotFoundError на 3.9.6» (тот же непрослеженный замер) для выбора JSON. Наследует
блокер ADR-0001 (закрывается тем же артефактом). Механика гейта (bd-задача + в конец
очереди) — не спайк-решение, а решение пользователя на гейте ТЗ (STATE стр. 30),
корректно атрибутировано «выбор пользователя 2026-07-15». Зачёт по этой половине.

**ADR-0003/0004 объявлены, но НЕ написаны.** design стр. 49: «ADR-0003 deny-стены (после
негативного спайка) · ADR-0004 гейт как bd-задача + сортировка в конец». ADR-0004
дублирует уже принятое в ADR-0002 (гейт+сортировка) — либо мёртвая ссылка, либо
рассинхрон нумерации. ADR-0003 (deny-стены) отложен «после негативного спайка» — логично
(F7 сначала доказывает механику), но тогда F7 обязан ПОРОДИТЬ ADR-0003 как артефакт
приёмки, а F7 этого не требует. МЕЛОЧЬ (нумерация) + вход в §7.

## 7. F7 негативный спайк стен = гейт входа в M3 — достаточно ли рано (reward hacking)?

**ЧАСТИЧНО. Гейт есть, но поставлен ПОЗЖЕ, чем требует логика reward hacking.**
F7 приёмка: «спайк не прошёл → эскалация ДО дальнейшей сборки M3»; проверка помечена
«гейт входа: этот тест ДО остальной сборки M3». Это соответствует ТЗ (стр. 119:
«негативный спайк = гейт входа в BUILD»). НО:

Проблема порядка: F7 стоит в **M3** (features стр. 104: M3 = F7-F8), тогда как воркеры
с bypass×$HOME спавнятся уже в **M1** (F1 запускает реальный claude в терминале с
глобальным settings.json = bypass). Между M1 и M3 (два майлстоуна, F1–F6) воркеры уже
исполняют реальные задачи под bypass БЕЗ доказанных стен. ТЗ формулирует спайк как «гейт
входа в BUILD» — а BUILD начинается на M1, не M3. Design/features сдвинули стену-гейт на
M3, фактически разрешив 6 фич реального автономного исполнения под непроверенным bypass.
Reward hacking / METR-риск (промпт-запреты бесполезны, только стены) реализуется ровно
в окне M1–M2, где стен ещё нет. Это ослабление условия ТЗ. **БЛОКЕР: негативный спайк
стен (F7) обязан быть гейтом входа в M1 (первый реальный спавн воркера), а не M3** —
либо явный ADR, почему M1–M2 гоняются на orc-test-проектах, где ущерб от пробоя стены
приемлем (но F1 hello.txt-проект тоже под bypass×$HOME, читает ~/.ssh при желании).
Как минимум — F1/F3/F6 должны спавнить воркера с УЖЕ сгенерированными deny-стенами
(F7-генератор), даже если полный негативный спайк формализуется в M3.

Гудхарт-проверка самого F7: «как пройти, не дав ценности?» — спайк из ДРУЖЕСТВЕННОЙ
интерактивной сессии (стены не тестируются реально). ТЗ закрыл это формулировкой
«из-под воркера» (R-brief р.3). F7 приёмка: «из-под воркера `git push`/`rm -rf`/чтение
~/.ssh ЗАБЛОКИРОВАНЫ» — «из-под воркера» сохранено. Зачёт по Гудхарту F7.
