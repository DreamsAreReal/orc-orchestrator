## 1. Ось «доказательства существования»: что проверено живьём, а что — только по докам

Инвентарь ставок архитектуры (RS-01, слои 1–3 + Mac-ops) против фактических проб:

| Объект / механика | Роль в архитектуре | Статус доказательства |
|---|---|---|
| beads/bd v1.1.0: init/create/dep/ready/claim/close | Слой 1 целиком | **ПРОБА PASS** (живьём, полный цикл, циклы блокируются) |
| osascript → Terminal tab → команда | Запуск воркера | **ПРОБА PASS** (но команда была `echo`, не `claude`) |
| claude CLI 2.1.193: agents/daemon/--bg/--worktree | Инфраструктура | **ПРОБА PASS** (уровень `--help`/detect, не исполнение) |
| ccusage blocks --active --json | Допуск/back-pressure | **ПРОБА PASS** (поля есть; происхождение «лимит ~271M» не установлено) |
| Keychain доступен | Предполётная проверка | **ПРОБА PASS, но из интерактивной GUI-сессии** — это НЕ проверка контекста LaunchAgent |
| **Happy (слой 3 целиком)** | Единственный канал к человеку | **ТОЛЬКО ДОКИ.** Не установлен, push round-trip не прогнан, ответ с телефона не проверен |
| **LaunchAgent (Aqua) + Keychain на практике** | Способ жизни диспетчера, риск №1 карты рисков | **ТОЛЬКО ДОКИ.** Ни одного plist не написано, `claude auth status` из LaunchAgent-контекста не запускался |
| **PostToolUse-хук → heartbeat** | Краеугольный камень watchdog | **ТОЛЬКО ДОКИ.** Хук не сконфигурирован, ни одного heartbeat-файла не существует, каденция не замерена |
| AskUserQuestion «ждёт в терминале», не жжёт токены | Механика гейта | **ТОЛЬКО ДОКИ** (и см. коллизию с RAM в §4.1) |
| caffeinate/pmset ночной прогон | Mac-ops | НЕ пробовано (допустимо, если контур объявлен дневным — но это решение не принято, см. §4.2) |
| Строки лимитов CLI (session/weekly/Opus) | Парсер back-pressure | **НИ ОДНОЙ ФИКСТУРЫ** с этой версии CLI на этом Mac |
| tmux как транспорт (Р10 пессимиста) | Альтернатива osascript | НЕ пробовано, противоречие не разрешено |
| dolt 2.1.10 (демон-зависимость beads) | Скрытая зависимость слоя 1 | Установлен, футпринт RAM **не замерен** — на машине с 282MB свободно |

Живая перепроверка (мной, 2026-07-15): repo slopus/happy жив (клиент для Claude Code,
E2E, push) — существование подтверждено. Но обнаружена незамеченная корпусом механика:
**взятие управления с телефона «перезапускает сессию в remote mode»** — влияние на
heartbeat/watchdog/терминал-вкладку не исследовано никем. Также по докам hooks существует
Notification-хук с матчерами типа permission_prompt/idle_prompt — потенциально ГОТОВЫЙ
внешний сигнал «сессия ждёт ввода» (детект парковки на гейте), который корпус упоминает
одной строкой в build-vs-buy и нигде не разбирает; требует пробы на 2.1.193.

**Вывод оси:** пробы покрыли лёгкое (bd, echo-в-терминал, JSON ccusage) и обошли все
три самых рискованных объекта: Happy, LaunchAgent+Keychain, heartbeat-хук. Гейт «проба
исполнимости PASS» пройден способом, не дающим ценности для главных рисков, — это
Гудхарт на пробу (принцип 4).

## 2. Ось «глубина»: вопросы сборки, на которые корпус НЕ отвечает

Д1. **Как ответ человека возвращается в припаркованную сессию?** RS-01 описывает ДВЕ
несовместимые механики гейта: (а) слой 1 — гейт = блокирующая bd-задача (воркер
освобождается, задача не в ready до закрытия человеком); (б) слой 3 — живая сессия
стоит в AskUserQuestion в терминале, ответ через Happy. Протокол round-trip
(парковка→push→вердикт→продолжение) заявлен в «пишем сами», но ресерч не дал ни
фактуры, ни выбора. Для design это вопрос №1.

Д2. **Improvement-половина mixed-класса пуста.** Патчи ~/.claude/skills/pipeline
(легализация docs/tasks/<слаг>/, развилка фазы 0) — в корпусе ноль ресерча: не
инвентаризованы места скилла, где зашиты раскладка workspace, discovery STATE.md,
пути хуков; нет характеризационного набора «что не должно сломаться» (принцип 15:
improvement без baseline = блокер). Пессимист Р4 описывает РИСК патчей, но не даёт
инвентаря точек изменения.

Д3. **Механика стен при глобальном bypass.** Пессимист требует sandbox/deny/allowlist,
RS-01 подтверждает существование sandbox — но никто не выяснил ДЛЯ ЭТОЙ УСТАНОВКИ:
можно ли per-project/per-session переопределить глобальный bypassPermissions из
settings.json; синтаксис deny-rules на 2.1.193; MCP-allowlist per-задача (в окружении
живут посторонние MCP — Glovo и пр., батч-воркеру они не нужны). Секцию ТЗ «границы
песочницы» писать не из чего.

Д4. **Детект статуса воркера снаружи** (работает / ждёт гейта / умер / завис):
`claude agents --json` обнаружен пробой, семантика не исследована; Notification-хук
не разобран. Диспетчер без этого слеп.

Д5. **Что именно кладётся в bd при закидке** (метка режима micro/lite/full, приоритет,
модель, уровень автономии) — три документа предлагают поля, синтеза нет.

Д6. Lease TTL/reaper: есть ли у bd v1.1.0 нативные lease-механизмы — не проверено
(заявлено «пишем сами» без проверки, что не изобретаем существующее).

## 3. Ось «спайки с цифрами»: чего не хватает для design-решений

Ц1. **RSS живого инстанса claude на ЭТОМ Mac.** Вывод «1 воркер» получен формулой
из чужих футпринтов (issues, другие машины). При 282MB свободных реальный вопрос —
«влезает ли вообще 1 воркер + dolt + Terminal + диспетчер без смерти в свопе», и он
не отвечен числом. Это порог осуществимости всего продукта.

Ц2. **Стоимость одной задачи через конвейер** (токены и минуты окна для micro/lite/full).
Без этого числа не существуют: прогноз «10 задач ≈ M окон», yield-management (идея 4),
интейк-нормировка (Р12 «оверхед превышает ценность»), бюджет/задачу (Р3). Ирония:
ccusage-проба показала burn rate 418k ток/мин ТЕКУЩЕЙ сессии — данные для замера были
под рукой, замер per-task не снят.

Ц3. **Каденция heartbeat** (медиана/хвост интервалов между PostToolUse на живой
конвейерной сессии, включая длинные thinking-паузы). Без распределения порог «нет
heartbeat N минут = завис» неопределим → watchdog будет убивать здоровые сессии либо
не ловить зависшие. Гейт watchdog сейчас НЕИЗМЕРИМ (принцип 14: обвязка — как фича).

Ц4. **Фикстуры строк лимитов** с CLI 2.1.193 (или хотя бы зафиксированные из офспеки
тексты + pin версии) — без них contract-тесты обвязки (лечение Р7) не построить.

Ц5. Латентность round-trip Happy (push → ответ → продолжение сессии) — ноль замеров.

## 4. Ось «противоречия», которые ТЗ обязано разрешить явно

П1. **Парковка на гейте × RAM=8GB (никем не замечено).** RS-01: «припаркованные на
гейте задачи токены не жгут» — верно про токены, но живая сессия в AskUserQuestion
держит 2.5–8GB RAM, т.е. ЕДИНСТВЕННЫЙ воркер-слот машины. На этом Mac «парковка в
терминале» = остановка всего контура до ответа человека. Либо гейт = bd-задача +
kill/fresh-restart после ответа (тогда Happy для гейтов не критичен, но нужен
чекпойнт «припаркован на гейте» в STATE задачи), либо живой AskUserQuestion (тогда
1 гейт блокирует смену). Выбор — в ТЗ, с ценой в единицах пропускной способности смены.

П2. **День vs ночь.** Цель продукта — «к утру готово»; RS-01 — caffeinate, ночные
прогоны; максималист — ночная смена как signature («газета», идея 11 «ночь чисто
машинная»); пессимист Р6 — «контур ДНЕВНОЙ, обещать за рабочий день, ночной режим —
отдельная волна». Прямой конфликт по главному обещанию. ТЗ обязано выбрать рамку v1.

П3. **Деградация Opus→Sonnet:** RS-01 — тихий штатный фолбэк back-pressure;
максималист (идея 5) — норма планирования (tiering по фазам); пессимист Р3 —
деградация = СОБЫТИЕ ГЕЙТА, не тихая. Три позиции, нужна одна.

П4. **Спекулятивное предисполнение (идея 3) против RAM-реальности:** экономика идеи
выведена из «окна сгорают впустую, парковка бесплатна» — на 1-воркерной машине
спекуляция стоит не токены, а единственный слот (и RAM). EV-аргумент максималиста на
этой машине неверен; в ТЗ — либо явное «P2 до апгрейда железа», либо пересчёт цены.

П5. **Метрика успеха:** RS-00 угол 3 — «задач доведено/сутки, доля без вмешательства»;
пессимист Р1 — DONE count оптимизируем без ценности (Гудхарт на весь продукт), мерить
«DONE, подтверждённый потреблением» + аудит N%. Метрика продукта в ТЗ — одна, и это
выбор между ними.

П6. **Watchdog-рестарт:** RS-01 — kill + свежий рестарт от STATE.md штатно; пессимист
Р2 — STATE может быть отравлен, рестарт ТОЛЬКО после внешней проверки пост-условий,
несовпадение = парковка; Р3 — cap рестартов K (петля kill-restart жжёт окно). Спека
watchdog в RS-01 этих поправок не содержит — ТЗ обязано их вшить или отклонить.

П7. **Транспорт терминалов:** osascript+Terminal (проба) vs tmux-транспорт (Р10:
терминал = view, процесс живёт в tmux; TCC-прогон заранее). Несовместимые основания
для F1 — нужен ADR.

П8. **Rubber-stamp vs inbox:** максималист (идея 2: 10 решений за 5 минут за кофе)
и пессимист (Р11: именно это и есть смерть гейта; кап батча ≤5, необратимое — вне
батча) тянут формат карточки в противоположные стороны. ТЗ фиксирует формат и капы.

## 5. Жанровая полнота ресерч-батча и деконструкция эталона

- Реестр покрытия волны A есть (61→55/6), брейнштормы покрывают углы широко —
  зачёт по методу.
- **Appendix RS-01 отсутствует в docs/research/** — STATE ссылается на «(+ appendix)»,
  файла в workspace нет: trace-to-original 55 находок разорван для будущих фаз.
- **Деконструкция эталона без артефактов:** Bernstein назван «ближайшим полным
  референсом» и донором паттернов (janitor, merge queue, ledger-resume), но в
  docs/research/refs/ нет НИ ОДНОГО добытого артефакта (скрин кабины cmux, формат
  ledger Bernstein, карточка Happy). По researcher.md п.6 деконструкция по памяти =
  ДЕГРАДИРОВАНО; design будет проектировать «газету»/кабину/карточку вслепую.

## 6. Цена среза (для решения на гейте)

| Срез | Цена в единицах ущерба потребителя контура |
|---|---|
| Не пробовать Happy round-trip | Первая смена: все задачи молча висят на гейте → нулевая ценность дня + подрыв доверия (анти-вау №1 по Р9) |
| Не пробовать LaunchAgent+Keychain | Контур не стартует без ручного запуска → «сам поднимается» из цели вычёркивается |
| Не мерить RSS инстанса на 8GB | Риск: своп/смерть машины посреди смены; неизвестно, осуществим ли продукт на этом железе вообще |
| Не мерить heartbeat-каденцию | Watchdog убивает здоровые сессии → задачи «падают» без вины, расход окна на рестарты (Р3) |
| Не снимать стоимость задачи | Обещание «10 задач к утру» не калибруемо; риск сжечь недельный кап за одну смену (угол 8) |
| Не инвентаризовать точки патча скилла | ТЗ improvement-половины пишется вслепую → риск Р4 (сломанный конвейер ломает все задачи дня) |

## Ответ на вопрос координатора: стоимость lite-задачи не замерена — блокер?

**НЕ блокер. Допустимое допущение при трёх условиях, фиксируемых в brief:**
1. В brief — явная секция «ДОПУЩЕНИЯ»: стоимость задачи неизвестна; снимается
   канарейкой (первая смена меряет себя) с датой/событием снятия.
2. До первого замера действует ЖЁСТКАЯ СТЕНА, а не прогноз: консервативный бюджет-кап
   на задачу и на смену в конфиге (упор в кап = парковка + событие в дайджест) — иначе
   первая же смена может сжечь weekly-кап (угол 8, Р3), а «канарейка меряет себя»
   превращается в оплату обучения недельной квотой.
3. ТЗ НЕ обещает «10 задач к утру» до калибровки — метрика первой смены:
   «замерено X задач, стоимость Y» (сама канарейка — деливерабл).
Обоснование: честный замер возможен только реальной задачей; риск ограничен внешней
стеной; это соответствует канону «волатильное — с пометкой срока годности».
