# ТЗ — Автономный таск-контур «orc» для Claude Code

## Цель
Существует контур: пользователь одной командой накидывает N задач по разным проектам → диспетчер на его Mac сам поднимает интерактивные инстансы Claude Code, гонит каждую задачу через конвейер pipeline до терминального статуса и по завершении смены отдаёт «газету» с гейт-карточками; успех = тестовая смена из 3 задач по 2 проектам доходит до конца без ручного вмешательства, кроме ответов на гейтах. (v1 — ДНЕВНАЯ смена; ночная — опция P1.)

## Контекст
Max x20 стабильно недоиспользуется; ручной запуск задач — узкое место. Research закрыт двумя волнами: 63 (волна S) + 55 (волна A, RS-01) подтверждённых первоисточниками находок + 7 проб на этом Mac (RS-02, .spikes/probe/): beads v1.1.0 работает, osascript-запуск работает, auth из LaunchAgent доказан (auth_exit=0), heartbeat-хук работает (медиана 316мс), ccusage отдаёт JSON-гейдж. Ключевое ограничение: **RAM 8 GB → 1 активный воркер** — контур = последовательная автономная смена, не параллельный рой (параллель проектов — P2 при апгрейде железа).

## Класс результата
mixed: software-service/CLI (greenfield: диспетчер `orc` + конфиги) + improvement (патчи ~/.claude/skills/pipeline — RS-02: 9 точек, 3 критических).

## Целевая планка
ГОТОВЫЙ ПРОДУКТ (ступень ≥3): контур можно доверить реальной смене без присмотра. Цена планки: ~4-5 майлстоунов + 1 майлстоун патчей конвейера (оценка из проб).

## Язык продукта
`UI language: en` (код, CLI-команды, README, коммиты — деливерабл-код). Пользовательские дайджест/уведомления/гейт-карточки — `ru`. Allowlist: beads/bd, Claude Code, ccusage, plist-ключи.
| Понятие | Канонический термин | Запрещённые синонимы |
|---|---|---|
| смена | shift | batch, run, session |
| задача | task (= bd issue) | ticket, job |
| гейт | gate | approval, checkpoint |
| диспетчер | dispatcher | orchestrator (занято конвейером), scheduler |
| воркер | worker | agent, instance |
| газета | shift report | digest, summary |

## Ставка вау
Ось ступени 4: **опыт «утренней газеты»** — проснулся → `orc status` → за минуту полная картина смены + гейт-карточки «что ты решаешь» (3 строки: скоуп/планка/полномочия) + git-тег смены. | Ставочные фичи: shift report, гейт-карточки, canary-предполёт. | Риск скуки: газета = простыня логов → контрход: первый экран ≤150 слов, решения — первыми строками, остальное по путям.

## Скоуп
### P0 (без этого задача провалена)
- `orc add <проект> "<текст>" [-p N]` (+ пачкой из stdin) — постановка 10 задач ≤5 мин.
- `orc status` — два режима: ЖИВАЯ смена (по строке на задачу: фаза/статус/минуты/расход; ждущие гейта — первыми; строка итога пула внизу) и итоговая ГАЗЕТА по завершении смены (первый экран ≤150 слов).
- Диспетчер-цикл: `bd ready --json` → claim → **re-validate при claim** (продуктовый слой проекта изменился после утверждения ТЗ-дельты задачи → мини-ре-валидация, не стройка на протухшем плане — Р5) → **пре-флайт воркера**: чистое git-дерево (грязное «не наше» → парковка «человек в репо?» — Р8) → **project-mutex** (одна активная задача на репо; 1 воркер глобально) → osascript-терминал: `cd <проект> && claude "<стартовый промпт>"`. Диспетчер ведёт PID-реестр, kill — только по своим PID.
- Стартовый промпт-шаблон: задача + workspace `docs/tasks/<слаг>/` + ссылка на продуктовый слой + вызов pipeline.
- Гейт-протокол: гейт = блокирующая bd-задача на человеке; механика сессии — по выбору на гейте ТЗ (вопрос 1); после утверждения — продолжение от STATE.md задачи. Гейт-карточка: скоуп/планка/полномочия (3 строки) + путь к полному ТЗ задачи + строка «цена ошибки»; НЕОБРАТИМЫЕ полномочия в батче не утверждаются — только отдельным вопросом; кап карточек в батче — конфиг.
- Back-pressure: детект строк лимитов CLI (session/weekly/Opus) → парковка до ресета; 429/529 → ретрай; допуск по `ccusage blocks --active --json` + порог свободной RAM.
- Watchdog: heartbeat из PostToolUse-хука; **различает «tool выполняется» (PreToolUse-маркер) от тишины**; порог = конфиг (калибрует канарейка); петля/тишина → внешняя проверка пост-условий (git/артефакты) → kill + свежий рестарт от STATE.md, cap рестартов = конфиг (дефолт 2) → эскалация в газету. Lease TTL: смерть воркера → задача возвращается в ready.
- Canary-предполёт перед сменой: bd, `claude auth status`, ccusage, канал уведомлений, свободная RAM, **спавн тестового терминала (osascript/TCC) из контекста LaunchAgent**; фейл = смена не стартует + уведомление.
- Бюджет-кап задачи и смены (токены из ccusage + wall-clock) в конфиге — ДО первого замера стоимости.
- `orc stop` (kill switch): всё останавливается ≤10 сек, задачи возвращаются в ready.
- Патчи конвейера: легализовать `docs/tasks/<слаг>/` (глобы pipeline-hooks.py ×2, scorecard, SKILL/phase-0 промпт-точки — RS-02) + развилка фазы 0 «STATE есть, но задача новая»; старое поведение не ломается.
- LaunchAgent (gui-сессия, абсолютный путь `/opt/homebrew/bin/claude`, PATH в plist) + конфиг `orc.toml` (все калибровки).
### P1 (важно)
- Уведомления на телефон: happy — ТОЛЬКО канонический npm-пакет `happy` (первоисточник: README slopus/happy; `happy-cli` архивирован 2026-02), после собственной пробы пейринга; либо Telegram-бот. До этого — macOS-уведомления.
- tmux-транспорт для ночного режима (App Nap/закрытое окно); ночная смена: caffeinate + политика «гейты днём, ночь машинная».
- Model-tiering политика per-фаза (Opus только на ТЗ/design/сложный build) — кратно больше задач за пул.
- Крэш-досье watchdog → кандидаты в failures.md конвейера (контур учится на провалах).
### P2 (желательно)
- Worktree-параллель непересекающихся задач одного проекта (по ADR); параллель проектов при RAM-апгрейде; yield-management окон; OTel-дашборд.

## Не входит в скоуп
- UI/kanban/веб-морда (уникальность — стыковка с конвейером, не интерфейс; Emdash/Conductor уже есть).
- Облачные Routines, GitHub Actions исполнение, поддержка не-Claude агентов, многопользовательность.
- Спекулятивное предисполнение веток гейта (экономика не сходится на 1 воркере — П4).

## Паспорт вкуса
**Референс-эталон**: bd CLI (краткие подкоманды, `--json` у всего, вывод grep-able) — деконструкция в RS-01/пробе beads; архитектурный референс — Bernstein (janitor-гейты, merge-queue, ledger-resume). Визуальные образцы/направление/токены: **N/A — нет UI** (CLI-продукт; «вкус» = дисциплина вывода). **Дисциплина вывода**: plain text, важное первой строкой, ≤80 колонок, без эмодзи (допустим статус-символ ✓/✗/⏸), без «Successfully!»-восклицаний; каждая команда имеет `--json`. **Signature-элемент**: «утренняя газета» — shift report с гейт-карточками и строкой итога смены («смена: 7 DONE, 2 ждут тебя, 1 упала; съедено 34% окна»). **Стек**: bash или python-stdlib (решит design по спайку), zero-dependency где возможно; конфиг TOML.

## Границы песочницы (стены, не промпт — Р13/Р14)
- Каждый воркер получает ПРОЕКТНЫЙ `.claude/settings.json` с deny-стенами: `git push` (все формы), `rm -rf` вне workspace проекта, чтение `~/.ssh` и чужих проектов, запись вне workspace; наследует PreToolUse-стены pipeline-hooks (секреты/force-push/no-verify).
- MCP: воркеры стартуют без MCP-серверов, кроме явного allowlist в `orc.toml` (дефолт — пустой).
- Env воркеров чистится от секрет-переменных по denylist конфига (прод-креды недоступны по построению).
- **bypassPermissions × весь $HOME** (глобальный settings.json действует на воркеров; protected paths в bypass РАЗРЕШЕНЫ — RS-01): оставить bypass + deny-стены выше ИЛИ переводить воркеров на `--permission-mode auto`+sandbox — вопрос анкеты гейта, решает пользователь.
- Оракулы приёмки (тесты/гейты задач) — read-only относительно воркера (reward hacking: промпт-запреты бесполезны, METR).
- **Негативный спайк стен = гейт входа в BUILD**: на pinned 2.1.193 живым спайком доказать, что deny-правила реально блокируют (git push, rm -rf вне workspace, чтение ~/.ssh из-под воркера) — enforcement имел историю полного отказа (anthropics/claude-code#6699); спайк не прошёл → архитектура стен пересматривается ДО сборки.

- Разрешено: создавать/менять файлы в workspace задач; git-коммиты и ветки в репо задач; создание папок новых проектов в согласованной директории; установка dev-инструментов через brew/npm ОФИЦИАЛЬНЫЕ пакеты; патчи ~/.claude — только диффом через гейт конвейера (этой задачи).

## Риски и допущения
- ДОПУЩЕНИЕ: стоимость lite-задачи в токенах неизвестна → бюджет-кап в конфиге ДО замера; «10 задач к утру» не обещаем до калибровки канарейкой; первая смена меряет себя.
- Reward hacking/галлюцинация успеха → DONE подтверждается внешними фактами (G1), оракулы вне записи воркера.
- Обновление claude ломает форматы → pin версии + self-test (G4-фикстуры) после апдейта.
- Rubber-stamp гейтов через неделю → карточки «что решаешь» + метрика времени-до-аппрува; полностью нерешаемо — честно.
- happy: официальный пакет не пробован (пробовался форк — снесён) → до собственной пробы канал = macOS-уведомления.
- Патч ~/.claude утром среды кладёт все задачи дня → патчи конвейера ставятся отдельным майлстоуном с канарейкой и откатом (git).
